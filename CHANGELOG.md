## v2.1 — Мультиязычность, режимы в UI/CLI, инженерные находки (в работе)

**Что сделали.**
- Выбор режима ридера в **CLI** (`--mode extractive|generative|hybrid`) и **Streamlit**
  (радио-переключатель + спиннер).
- Пресет `multilingual_config()` (эмбеддер `paraphrase-multilingual-MiniLM-L12-v2`) для EN+RU;
  RU-документ для демо банка (`demo/bank_products_ru.txt`).
- Рефактор reader'а: логика извлечения span вынесена в переиспользуемый `_extract`.

**Инженерная находка (для защиты).** Пробовали батчить reader (один forward на k фрагментов).
На CPU это **не ускорило**, а слегка замедлило: паддинг до самого длинного фрагмента добавляет
«пустой» compute. Вернулись к пофрагментной обработке (нативная длина). Вывод: батчинг ридера
выгоден на GPU, не на CPU — важно для on-premise на обычном железе.

**Статус.** Модель Qwen 2.5 3B (GGUF) перенесена в `models/` проекта — генеративный режим
запускается без настройки env; проверено на EN (`«2017 [1] [2]»`). **RU полностью заработал**:
многоязычный эмбеддер + Qwen дают многофактовый ответ с цитатами — на вопрос про банк-документ
получили `«Минимальный взнос — 500 рублей. Ставка по кредиту от 14,9%. [1] [2]»` (собрано из
двух фрагментов — то, что extractive не умеет).

**Находки при подключении RU (для защиты — «грабли и как чинили»).**
- Нужен пакет `sentencepiece` (токенизатор XLM-R) — иначе `Unrecognized processing class`.
- `sentence-transformers` 5.6.0 ломает офлайн-загрузку многоязычной модели (грузит через
  AutoProcessor). Обход: `local_files_only=True` при загрузке эмбеддера + не выставлять
  `HF_HUB_OFFLINE=1`; правильный фикс — пин `sentence-transformers<5` (в requirements).
- Экстрактивный `tinyroberta-squad2` — англоязычный: русский не читает. Вывод: для RU —
  режим `generative` (что дополнительно оправдывает двухридерную архитектуру).
- Скачивание: `scripts/download_models.py` (в venv нет консольной `huggingface-cli`).

## v2 — Генеративный ридер (Qwen) + защищаемая оценка

**Мотив.** Партнёр практики — региональный банк (куратор из ML-отдела). Цель сместилась с
«учебного прототипа» к **самодостаточному on-premise RAG-ассистенту по документам** с
доказанной точностью и бизнес-упаковкой. Ключевой аргумент для банка: всё локально —
ни один документ/вопрос не покидает периметр.

**Что сделали.**
- **Генеративный ридер** `generator.py` на локальной **Qwen 2.5 3B Instruct Q4_K_M** (GGUF,
  llama-cpp, офлайн). Grounded-промпт: отвечать строго по фрагментам, цитировать `[n]`,
  возвращать маркер отказа при отсутствии ответа. Валидировано на **EN и RU**.
- **Три режима ридера** в `pipeline.py` (выбирает пользователь):
  `extractive` (BERT), `generative` (Qwen), `hybrid` (Qwen отвечает + BERT проверяет
  обоснованность → анти-галлюцинационный guardrail).
- Ленивая загрузка ридеров (Qwen грузится только когда нужен).

**Оценка — устранили методологические дыры (P0).**
- **Раздельные calibration/test сплиты** для калибровки порога τ (была утечка: раньше τ
  подбирался и отчитывался на одной выборке → цифры завышены).
- **Oracle-baseline**: reader на золотом контексте — измеряет потолок и потерю на retrieval.

**Что выяснили (честные числа на held-out test).**
- retrieval: EM=73.6, **F1=75.8**, NoAns_F1=84.6.
- oracle-контекст: EM=79.1, **F1=81.0**.
- **Потеря на retrieval = 5.26 F1** (раньше заявляли «≈ оракул» без доказательства).
- Прежние 78.3 F1 были завышены утечкой на ~2.5 пункта.
- Oracle F1=81.0 совпал с эталонным dev-F1 tinyroberta-squad2 → реализация reader'а корректна.

**Найденные и исправленные баги по ходу.**
- Qwen иногда отвечал одними скобками `[2]` без текста — починили промптом (формат
  «ответ + цитата в конце», явный запрет пустого ответа).

## v1 — Экстрактивный RAG-baseline

**Мотив.** Задание практики: чат-бот по документу (SQuAD v2) с обработкой «нет ответа» и
доказанной точностью (EM/F1). Название проекта намекает на BERT-ридер.

**Что сделали.**
- Полный RAG-пайплайн на CPU: ingest (SQuAD/PDF/TXT) → чанкинг (по предложениям + overlap,
  размер в токенах) → эмбеддинги MiniLM (кэш) → FAISS (cosine) → retrieval top-k →
  экстрактивный reader `tinyroberta-squad2` (span + null-score) → двухуровневая абстенция →
  ответ + цитата.
- Интерфейсы: CLI (`chat`/`ask`) и Streamlit.
- Оценка: retrieval (Recall@k, MRR), end-to-end (EM/F1, HasAns/NoAns), калибровка τ,
  ablations по chunk_size × top_k. Артефакты в `eval/results/` (CSV+PNG).

**Что выяснили.**
- Retrieval сильный: Recall@5 ≈ 0.94, MRR ≈ 0.86.
- Ablation: F1 насыщается к top_k=3–5, латентность растёт линейно → выбрали k=5.
- **Важное ограничение**: на SQuAD контексты короткие (медиана 141 токен), поэтому при
  chunk≥256 чанкинг почти не срабатывает (256: режется 8% док., 512: 0.3%). Значит вывод про
  размер чанка **не переносится** на реальные длинные PDF — честно отражено в анализе.

**Баги, пойманные тестами/прогонами.**
- Бесконечный цикл в чанкере на длинном предложении без пунктуации (overlap не давал
  продвинуться) → переписали упаковку на гарантированно завершающийся for-цикл.
- Двойной счёт `recall@1` в ablations при `top_k=1` (`ks=(1,1)`) → дедуп ks.
- Конфликт TF/Keras 3 при импорте sentence-transformers → `USE_TF=0` в `__init__.py`.
