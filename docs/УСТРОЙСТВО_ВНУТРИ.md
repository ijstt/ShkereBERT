# Как ShkereBERT устроен внутри (инженерный уровень)

Это «второй этаж» над `ПОСОБИЕ.md`: там — что такое RAG и зачем каждый шаг, здесь — **как
именно** каждый шаг реализован, какие формулы и решения внутри, и почему сделано так.
Полезно перечитать перед предзащитой: вопросы «а как конкретно у вас…» закрываются отсюда.

---

## 1. Путь одного вопроса через систему

Возьмём реальный пример: `./run.sh ask "When was the transformer architecture introduced?"`.

1. **Индекс уже построен** (при старте): документ разрезан на чанки, каждый чанк — вектор
   в FAISS. Это происходит один раз, дальше вопросы обрабатываются по построенному индексу.
2. **Вопрос кодируется** тем же эмбеддером, что и чанки (`embeddings.py`) → вектор 384 чисел.
3. **FAISS ищет top-k=5** ближайших чанков по косинусной близости (`index.py`, `retriever.py`).
4. **Абстенция уровня 1** (`pipeline.py:85`): если похожесть лучшего чанка ниже
   `min_score` — сразу «нет ответа», ридер даже не запускается. Это отсекает вопросы
   «не по документу» (про них поиск не находит ничего похожего).
5. **Ридер** (по выбранному режиму) читает top-k чанков и решает: ответ + источник, либо
   **абстенция уровня 2** («в документе нет ответа», хотя тема близкая).
6. Пользователь получает: текст ответа, режим, причину решения (`reason` — прозрачность),
   список источников с пометкой ★ у тех, откуда взят ответ.

Двухуровневая абстенция — осознанное решение: retrieval-уровень ловит «вопрос вообще не о
том», reader-уровень ловит «тема та, но конкретного факта в документе нет». Это разные
режимы ошибок, и им нужны разные механизмы.

## 2. Чанкинг: жадная упаковка предложений с overlap (`chunking.py`)

**Единица измерения — токены эмбеддера, не символы.** Модель видит токены; чанк «в 1000
символов» может оказаться и 200, и 400 токенов в зависимости от языка и слов. Поэтому
`TokenCounter` считает длину тем же токенизатором, что у эмбеддера.

**Алгоритм (стратегия `sentence`, дефолт):**
1. Текст режется на предложения регэкспом по границам `.!?` (+ кавычки/скобки после).
2. Предложение длиннее `size` сразу дробится скользящим окном по токенам (fallback).
3. Предложения жадно упаковываются в чанк, пока влезают в `size` (256 токенов по умолчанию).
4. При переполнении чанк закрывается, и следующий начинается с **overlap-хвоста** —
   последних предложений на ~64 токена. Так факт, стоящий на границе, целиком попадает
   хотя бы в один из двух соседних чанков.

**Почему по предложениям, а не окном по токенам:** окно режет посреди фразы — и retrieval
получает обрывки без смысла, и extractive-ридер может выдать обрезанный span. Стратегия
`fixed` (чистое окно) оставлена в конфиге для ablation-сравнения.

**Грабли, которые мы прошли:** первая версия упаковки была на while-цикле и зацикливалась
на длинном предложении без пунктуации (overlap-хвост не давал продвинуться). Переписали
на for-цикл, где каждое предложение добавляется ровно один раз — завершение гарантировано
конструкцией, а не удачей. Поймано тестом `tests/test_chunking.py`.

## 3. Эмбеддинги и индекс: cosine через inner product (`embeddings.py`, `index.py`)

- Эмбеддер — `all-MiniLM-L6-v2` (EN, 384d) или `paraphrase-multilingual-MiniLM-L12-v2`
  (EN+RU, переключается env `SHKEREBERT_EMBED`). Оба лёгкие, живут на CPU.
- Векторы **L2-нормируются** → скалярное произведение равно косинусу угла. Поэтому индекс
  `IndexFlatIP` (inner product) фактически ищет по косинусной близости.
- `IndexFlatIP` — **точный** перебор, не приближённый. На нашем масштабе (тысячи–десятки
  тысяч чанков) поиск занимает миллисекунды, а точность оценки важнее. IVF/HNSW — это
  ответ на вопрос «а если документов миллион», и честный ответ: «заменить одну строку,
  FAISS это поддерживает».
- **Дисковый кэш эмбеддингов**: ключ = SHA1(модель + параметры чанкинга + список id чанков)
  (`embeddings.py:_corpus_signature`). Смена любого параметра меняет ключ — старый кэш не
  подхватится «незаметно». На CPU эмбеддинг корпуса — самая дорогая операция построения,
  кэш делает повторные прогоны ablations почти бесплатными.

## 4. Экстрактивный ридер: логиты span'ов и null-score (`reader.py`)

`deepset/tinyroberta-squad2` — RoBERTa, дообученная на SQuAD v2. На вход — пара
(вопрос, чанк), на выход — два вектора логитов: `start_logits` и `end_logits` по токенам.

**Извлечение ответа** (`reader.py:_extract`):
- `best_span_score = max(start_logits[i] + end_logits[j])` по валидным парам i≤j внутри
  контекста (не вопроса!), длиной ≤ `max_answer_len`. Перебираем не все пары, а топ-20
  кандидатов по start и по end — стандартная оптимизация из эталонного инференса SQuAD.
- `null_score = start_logits[CLS] + end_logits[CLS]` — «оценка пустого ответа». Модель
  SQuAD v2 обучена ставить CLS-логиты высоко, когда ответа нет.

**Решение об абстенции:** `gap = null_score - best_span_score`. Если `gap > τ` — «нет
ответа». τ — единственный свободный параметр решения, и он **не подбирается на глаз**:
калибруется перебором по сетке на calibration-сплите (максимум F1), затем метрики честно
считаются на отложенном test (`eval/eval_e2e.py`). Сейчас τ* ≈ −6.3.

Интуиция знака: τ < 0 означает «отвечаем даже когда null-score немного выше span-score» —
модель на SQuAD v2 систематически завышает null, и калибровка это компенсирует.

**Мультичанк:** ридер читает каждый из top-k чанков независимо, ответы сортируются по
`best_span_score`. Берём лучший. Каноническая агрегация null-скоров по чанкам — известное
улучшение, лежит в пуле B.

**Почему без батчинга:** пробовали один forward на все k чанков — на CPU стало *медленнее*:
паддинг до самого длинного чанка добавляет пустой compute. На GPU батчинг выгоден, на CPU —
нет. Замерено, вывод зафиксирован в CHANGELOG (v2.1).

## 5. Генеративный ридер: grounded-промпт для Qwen (`generator.py`)

Локальная **Qwen 2.5 3B Instruct**, квантизация Q4_K_M (GGUF, ~2 ГБ), движок llama-cpp,
`temperature=0` — детерминированность для воспроизводимой оценки.

Промпт-контракт (5 правил в system-prompt):
1. отвечать **только** по пронумерованным фрагментам;
2. формат «ответ + номера фрагментов в конце»: `«Столица — Париж [3].»`;
3. запрет отвечать одними скобками (реальный баг: модель отвечала `[2]` без текста —
   починили именно этим пунктом);
4. если ответа нет — вернуть ровно маркер `НЕТ ОТВЕТА` (это абстенция уровня 2 для
   генеративного режима);
5. отвечать кратко и на языке вопроса (поэтому один промпт работает для EN и RU).

Постобработка: регэкспом достаём `[n]`, валидируем диапазон, мапим на id чанков — так
ответ получает **машиночитаемые ссылки на источники**, а не просто текст.

Ограничение честно признаём: маркер отказа — это «доверься модели», в отличие от
калиброванного τ у extractive. Поэтому качество абстенции генератора измеряется отдельно
(`eval/eval_generative.py`), а в hybrid-режиме генератор подстрахован экстрактивным ридером.

## 6. Hybrid: генерация с верификацией (`pipeline.py:_answer_hybrid`)

Идея: Qwen отвечает красиво, но может галлюцинировать; BERT отвечает скупо, но его
уверенность калибрована. Скрещиваем:

| Qwen | BERT (gap ≤ τ?) | Итог |
|---|---|---|
| ответил | подтверждает | ответ Qwen, пометка **verified by extractive** |
| ответил | не подтверждает | ответ Qwen, пометка **UNVERIFIED (возможна галлюцинация)** |
| отказался | нашёл ответ | **fallback на extractive** (span-ответ) |
| отказался | не нашёл | «нет ответа» (both abstained) |

Это дешёвый faithfulness-контроль без третьей модели: верификатор уже есть в системе.
Для банковского сценария пометка UNVERIFIED — сигнал «проверь руками», что снижает цену
ошибки без потери удобства генеративных ответов.

## 7. Оценка: почему цифрам можно верить (`eval/`)

Методологический каркас (это самое защищаемое место проекта):

- **Официальная метрика** `squad_v2` из HF `evaluate` — не самописный F1.
- **Calibration/test split**: τ подбирается на одной половине, метрики отчитываются на
  другой. В ранней версии этого не было — и цифры были завышены на ~2.5 F1. Мы это
  обнаружили, починили и честно записали в CHANGELOG (сильная история для защиты).
- **Oracle-baseline**: ридер на золотом контексте = потолок ридера. Разница
  oracle − retrieval = **цена этапа поиска** (у нас ≈ 5.3 F1). Без oracle фраза «retrieval
  почти не теряет» была бы голословной.
- **Сырые предсказания собираются один раз**, свип τ идёт по сохранённым gap'ам — модель
  не перезапускается 31 раз (`eval_e2e.py`).
- **Ablations** (`ablations.py`): chunk_size × top_k на одном наборе; индекс переиспользуется
  внутри одного chunk_size (меняется только k) — свип дешёвый.
- **Длинные документы** (`eval_longdoc.py`): SQuAD-контексты короткие (медиана 141 токен),
  на них chunk_size≥256 просто не срабатывает. Мы склеиваем контексты статьи (title) в
  документы по ~5–6k токенов — там чанкинг работает по-настоящему, и появляется отдельная
  метрика `answer-recall@k` (top-k чанков содержат эталон строкой), которая ловит в том
  числе разрезание ответа границей чанка.
- **Multi-seed CI** (`eval_multiseed.py`): весь протокол повторяется на 5 сидах выборки →
  mean ± 95% CI. Отвечает на вопрос «а не повезло ли вам с выборкой».
- **Сравнение режимов** (`eval_generative.py`): extractive/generative/hybrid через
  продуктовый `RAGPipeline.answer` на одной выборке. Для генерации, кроме строгого F1
  (который штрафует многословие), считается **gold-containment** — содержится ли эталон
  в ответе после нормализации.

## 8. Конфигурация и инженерные решения (`config.py` и вокруг)

- Все параметры — в одном месте, frozen dataclasses. Ablations переопределяют их через
  `dataclasses.replace`, не трогая глобальное состояние.
- **Ленивая загрузка**: Qwen (~2 ГБ RAM) грузится только при первом обращении к
  генеративному/hybrid режиму (`pipeline.py:@property generator`). Extractive-пользователь
  не платит за LLM ничего.
- `lru_cache` на загрузке моделей: повторное создание Reader/Embedder/LLM с теми же
  параметрами переиспользует уже загруженную модель.
- Переключатели через env: `SHKEREBERT_EMBED` (эмбеддер), `SHKEREBERT_LLM` (путь к GGUF),
  `SHKEREBERT_DATA` (кэш) — удобно для демо и ablations без правки кода.
- `USE_TF=0 USE_FLAX=0` в `shkerebert/__init__.py` — фикс конфликта Keras 3 с
  sentence-transformers (реальные полевые грабли, см. CHANGELOG).
- Полностью офлайн: модели в локальном HF-кэше и `models/`, датасет закэширован;
  прогоны идут с `HF_DATASETS_OFFLINE=1` — сеть не нужна вообще (on-premise аргумент
  подтверждён буквально).

## 9. Известные ограничения (говорить о них самим — сильный ход)

1. **Extractive-ридер только EN** (tinyroberta-squad2). Для RU — generative/hybrid;
   это дополнительный аргумент двухридерной архитектуры, но и ограничение.
2. **Абстенция генератора не калибрована** — это промпт-контракт, а не порог на скоре.
   Измеряем её качество отдельно; в hybrid подстраховано верификатором.
3. **Один непрерывный span у extractive** — составной ответ из двух мест документа он не
   соберёт (генеративный собирает, проверено на RU-демо банка).
4. **PDF без текстового слоя** (скан) требует OCR — вне текущего скоупа.
5. **Латентность генерации на CPU** — секунды на вопрос (точные цифры в
   `eval/results/generative_eval.csv`). Extractive — сотни миллисекунд.
6. **SQuAD ≠ банковские документы**: доменный сдвиг (таблицы, юридические формулировки)
   потребует своей разметки для калибровки τ и, возможно, дообучения эмбеддера.
