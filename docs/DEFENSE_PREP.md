## 3. Речь по слайдам (что говорить)

### Слайд 1–2: Задача

> «Наша задача — построить чат-бот, который отвечает на вопросы по текстовому документу. Три ключевых требования: (1) ответ должен быть основан на документе, а не выдуман; (2) если ответа нет — система честно об этом говорит; (3) показываем источник — фрагмент, из которого взят ответ. Качество оцениваем по метрикам Exact Match и F1 на датасете SQuAD v2.»

### Слайд 3: Архитектура

> «Вот как это работает. Документ режется на фрагменты по 256 токенов. Каждый фрагмент превращается в вектор — эмбеддинг. Все векторы складываются в FAISS-индекс. Когда приходит вопрос, мы тоже превращаем его в вектор и ищем 5 ближайших фрагментов. Дальше ридер извлекает или генерирует ответ.»

### Слайд 4: Датасет

> «SQuAD v2 — это 150 тысяч вопросов по Wikipedia. Уникальная особенность — 50 тысяч вопросов без ответа, специально составленных так, чтобы обмануть модель. Это позволяет оценить, как система обрабатывает случай "нет ответа". Проблема данных: контексты короткие — медиана 141 токен. Поэтому мы дополнительно тестировали на длинных документах и разделили данные на calibration и test, чтобы не было утечки.»

### Слайд 5: Модели

> «Мы используем три предобученные модели. MiniLM — маленькая, 22 миллиона параметров — превращает текст в вектор для поиска. TinyRoBERTa — 80 миллионов — выделяет кусок текста с ответом. Qwen — 3 миллиарда — генерирует ответ своими словами. Мы их не обучали — это transfer learning: берём готовое предобучение и строим пайплайн вокруг. Всё работает на CPU, офлайн — данные не покидают периметр.»

### Слайд 6: Три режима

> «Extractive — BERT вырезает span из текста. Быстро, 50 миллисекунд, точно. Но только один непрерывный кусок. Generative — Qwen генерирует ответ с цитатами, может собрать из нескольких фрагментов, работает на русском. Но медленнее и может галлюцинировать. Hybrid — Qwen отвечает, а BERT проверяет: есть ли подтверждение в тексте. Это наш guardrail — перехватывает 70% галлюцинаций.»

### Слайд 7: Метрики Retrieval

> «Recall@5 = 94% — в 94 случаях из 100 нужный фрагмент оказывается в пятёрке лучших. MRR = 0.86 — правильный фрагмент обычно между первой и второй позицией. Вот график: [показать ablation_recall_topk.png]. Видно, что после k=5 recall растёт минимально, а латентность — линейно. Поэтому k=5 — оптимум.»

### Слайд 8: Метрики E2E

> «End-to-end F1 = 75.8 — на held-out test, без утечки. Oracle-baseline (если retriever всегда находит нужный фрагмент) даёт F1 = 81. Разница 5.2 пункта — это потеря на этапе поиска. Oracle совпал с эталонным F1 модели — значит наша реализация корректна. Вот кривая калибровки порога: [показать e2e_threshold_curve.png]. Красная линия — выбранный порог tau.»

### Слайд 9: Ablation

> «Мы проверили влияние размера чанка и top_k. Вот график: [показать ablation_f1_topk.png]. F1 насыщается к k=3–5 для всех размеров. На SQuAD разница chunk_size минимальна, потому что контексты короткие. Но на длинных документах chunk=64 выиграл 5 пунктов F1 у chunk=256 — это значимый эффект.»

### Слайд 10: Стабильность

> «Чтобы убедиться, что результат не случайность, мы прогнали оценку на 5 разных seed'ах. F1 = 76.9 плюс-минус 3.9. Наш основной результат 75.8 — внутри доверительного интервала. Разница конфигураций в 1–2 F1 — это шум.»

### Слайд 11: Демо

> «Вот наш интерфейс — Streamlit. [Показать скриншот.] Можно загрузить свой PDF или использовать SQuAD. Задаём вопрос — получаем ответ с цитатой и уверенностью. Можно переключать режим ридера.»

### Слайд 12: Итоги

> «Что сделали: RAG-пайплайн с тремя режимами, защитой от галлюцинаций и честной оценкой. Всё офлайн на CPU. Что можно улучшить: русский экстрактивный ридер, GPU для ускорения Qwen, OCR для PDF-сканов.»

## 4. Ответы на вопросы куратора

### «Что за задача? Что нужно было сделать?»
> Чат-бот по документу: пользователь задаёт вопрос, система находит ответ в тексте, показывает источник. Если ответа нет — честно говорит. Оценка по EM/F1 на SQuAD v2.

### «Вы обучали модель?»
> Нет. Мы используем три предобученные модели (transfer learning). Наша работа — пайплайн: чанкинг, индекс, retrieval, абстенция, калибровка, hybrid-верификация, eval-скрипты, интерфейсы.

### «Почему именно эти модели?»
> MiniLM — компактный эмбеддер (22M), работает на CPU за миллисекунды. TinyRoBERTa — лёгкая QA-модель, обучена на SQuAD v2 с поддержкой «нет ответа». Qwen 3B — генеративная, работает локально (GGUF), не требует GPU и облака.

### «Чем модели отличаются друг от друга?»
> MiniLM — bi-encoder: кодирует текст в вектор, ищет похожие. TinyRoBERTa — extractive reader: вырезает span ответа из текста. Qwen — generative: создаёт новый текст на основе контекста. Поиск → чтение → генерация — три разные задачи.

### «Какая модель лучше справилась?»
> Для точности: extractive (F1 75.8, 50 мс). Для естественности и русского: generative (собирает из нескольких фрагментов, но F1 ниже из-за многословия). Для надёжности: hybrid (Qwen + проверка BERT, ловит 70% галлюцинаций).

### «С какими проблемами столкнулись?»
> (1) Утечка при калибровке tau — подбирали и оценивали на одних данных, F1 был завышен на 2.5 пункта. Починили разделением calibration/test. (2) Qwen галлюцинировал на вопросах без ответа — решили промптом + hybrid-верификацией. (3) Бесконечный цикл в чанкере — переписали с гарантией прогресса. (4) Батчинг на CPU не ускорил — замерили, отказались.

### «Что значит F1 = 75.8?»
> Token-level F1: сравниваем слова предсказания и эталона. Precision — сколько из предсказанных слов правильные. Recall — сколько из правильных слов предсказано. F1 — их гармоническое среднее. 75.8 значит в среднем 75.8% совпадения токенов с эталоном.

### «Что такое Recall@5 = 0.94?»
> В 94% случаев правильный фрагмент оказался среди 5 лучших найденных. Это значит, что наш поиск почти всегда находит нужный кусок документа.

### «Почему tau = −6.3?»
> Порог подобран на calibration-сплите как argmax F1. Если gap (null_score − span_score) > tau, система говорит «нет ответа». tau зависит от конфигурации и домена — его нужно калибровать заново при смене параметров.

### «Какие ограничения?»
> (1) BERT — только один непрерывный span. (2) Метрики на SQuAD (энциклопедический) — на узких доменах нужна перекалибровка. (3) Qwen на CPU медленный (10 с). (4) PDF — только текст (нет OCR). (5) Русский — только generative.

### «Что бы улучшили?»
> Русский ридер (xlm-roberta), GPU для Qwen, OCR для сканов, streaming-генерацию, авто-калибровку tau при смене домена.

### «Как работает для русского?»
> Меняем эмбеддер на мультиязычный (paraphrase-multilingual-MiniLM). Промпт Qwen автоматически переключается на русский по наличию кириллицы. Экстрактивный BERT не работает на русском (обучен на EN). Для русского рекомендуем generative-режим.

### «Как вы поняли, что результат не случайность?»
> Multi-seed оценка: 5 разных подвыборок, F1 = 76.9 ± 3.9 (95% CI). Наш результат 75.8 внутри интервала. Разница конфигураций в 1–2 F1 — шум выборки, доверяем только эффектам крупнее 4 F1.

### «Зачем oracle-baseline?»
> Показывает потолок ридера (F1 81.0). Разница с нашим пайплайном (75.8) = 5.2 — это потеря на retrieval. Если бы потеря была 20 — проблема в поиске. У нас 5.2 — поиск хороший, основное ограничение — сам ридер.
