# Пособие по технологиям ShkereBERT

Документ для команды: объясняет, из чего состоит проект, какие технологии используются на
каждом шаге, зачем они нужны и где это в коде. Читается сверху вниз; в конце — глоссарий.

---

## 0. Что вообще делает система (в двух предложениях)

Пользователь задаёт вопрос по документу(ам). Система находит нужные фрагменты текста и
формирует ответ **строго по ним**, с ссылкой на источник, а если ответа в документах нет —
честно говорит «нет ответа». Это называется **RAG** (Retrieval-Augmented Generation).

---

## 1. Что такое RAG и зачем он (а не «просто LLM»)

Если задать вопрос обычной языковой модели, она отвечает «из головы» — и может **выдумать**
факт (галлюцинация), особенно про ваши внутренние документы, которых она не видела.

**RAG** решает это в два шага:
1. **Retrieval (поиск)** — найти в базе документов фрагменты, релевантные вопросу.
2. **Generation/Reading (ответ)** — сформировать ответ, опираясь **только** на эти фрагменты.

Плюсы для бизнеса (например банка):
- ответы основаны на **актуальных внутренних документах**, а не на «памяти» модели;
- можно показать **источник** (цитату) — важно для доверия и комплаенса;
- всё работает **локально** (on-premise) — данные не уходят наружу.

## 2. Архитектура и поток данных

```
Документы (SQuAD / PDF / TXT)
        │  ingest.py          → превращаем в текст
        ▼
   Чанкинг                    chunking.py    → режем на фрагменты
        │
        ▼
   Эмбеддинги                 embeddings.py  → текст → вектор (числа)
        │
        ▼
   Векторный индекс (FAISS)   index.py       → быстрый поиск по векторам
        │
        ▼  вопрос ─► retriever.py            → top-k релевантных фрагментов
        ▼
   Ридер (на выбор):          reader.py / generator.py
     • extractive — BERT выделяет точную фразу-ответ (span)
     • generative — Qwen пишет естественный ответ по фрагментам
     • hybrid     — Qwen отвечает, BERT проверяет обоснованность
        │           pipeline.py — связывает всё + логика «нет ответа»
        ▼
   Ответ + confidence + ЦИТАТА  → CLI (cli.py) / веб (app/streamlit_app.py)
```

Каждый модуль — отдельный файл в `shkerebert/`. Всё настраивается через `config.py`.

## 3. Технологии по шагам

### 3.1 Ingestion — загрузка документов (`ingest.py`)
Приводим любой источник к единому виду `Document` (текст + метаданные):
- **SQuAD v2** — учебный датасет вопросов-ответов по абзацам Википедии (для замеров качества);
- **PDF** — извлекаем текст библиотекой **pypdf** (у сканов без текстового слоя нужен OCR);
- **TXT** — просто читаем файл.

### 3.2 Чанкинг — деление на фрагменты (`chunking.py`)
Зачем резать документ:
- ридер видит ограниченное «окно» текста;
- поиск точнее работает на **коротких смысловых кусках**, чем на всём документе.

Как у нас:
- режем **по предложениям** и упаковываем их в чанки заданного размера **в токенах**
  (токен ≈ кусочек слова — то, что реально «видит» модель, а не символы);
- делаем **overlap** (перекрытие соседних чанков), чтобы ответ не «разрезался» на границе.

Тонкость (наш вывод): у SQuAD абзацы короткие (медиана 141 токен), поэтому при размере
чанка 256+ деление почти не срабатывает. На реальных длинных PDF чанкинг важнее.

### 3.3 Эмбеддинги — текст в вектор (`embeddings.py`)
**Эмбеддинг** — это представление текста числовым вектором (у нас 384 числа) так, что
близкие по смыслу тексты дают близкие векторы. Тогда «похожесть по смыслу» = близость векторов.

- Модель: **sentence-transformers** (`all-MiniLM-L6-v2` для EN, лёгкая и быстрая на CPU;
  `paraphrase-multilingual-MiniLM-L12-v2` для EN+RU).
- Меру близости берём **косинусную** (нормируем векторы → скалярное произведение = косинус).
- Эмбеддинги считаются один раз и **кэшируются на диск** — на CPU это самая дорогая операция.

### 3.4 Векторный индекс — быстрый поиск (`index.py`)
**FAISS** — библиотека для поиска ближайших векторов. Используем `IndexFlatIP` — точный
перебор по всем векторам (корпус небольшой, точность важнее скорости). Индекс сохраняется
на диск. Для очень больших баз есть приближённые индексы (IVF/HNSW) — это на будущее.

### 3.5 Retrieval — поиск релевантных фрагментов (`retriever.py`)
Кодируем вопрос в такой же вектор и берём **top-k** ближайших чанков (dense-поиск).
Опционально (план):
- **reranker** (кросс-энкодер) — переоценивает кандидатов, глядя на пару (вопрос, фрагмент)
  целиком; точнее, но медленнее;
- **hybrid BM25** — добавить классический словарный поиск к смысловому.

### 3.6 Экстрактивный ридер — BERT выделяет ответ (`reader.py`)
**BERT** (у нас `deepset/tinyroberta-squad2`) — модель, обученная на SQuAD v2. Она берёт
(вопрос + фрагмент) и предсказывает **начало и конец** ответа внутри фрагмента (span).
- Технически модель выдаёт «логиты» начала и конца для каждого токена; мы ищем пару
  (start ≤ end) с максимальной суммой — это и есть ответ.
- Плюс: ответ — **дословная цитата** из документа, легко считать метрики EM/F1.
- Минус: не умеет «собрать» составной ответ (вернёт один непрерывный кусок).

### 3.7 Обработка «нет ответа» и калибровка порога τ
SQuAD v2 содержит вопросы **без ответа** — система обязана это распознавать.
- Модель выдаёт **null-score** — «оценку пустого ответа». Если пустой ответ «сильнее»
  лучшего span'а (с запасом), считаем, что ответа нет.
- Запас регулируется **порогом τ**. Его **калибруют**: перебирают значения на одной части
  данных (calibration) и берут то, что даёт максимум F1; проверяют на **другой** части
  (test). Так избегаем «подгонки под тест» (утечки данных).
- Наш график `eval/results/e2e_threshold_curve.png` показывает компромисс: слишком строгий τ
  → отказываемся отвечать даже когда ответ есть; слишком мягкий → отвечаем там, где не надо.

### 3.8 Генеративный ридер — Qwen (`generator.py`)
**Qwen 2.5 3B Instruct** — локальная языковая модель. Формирует **естественный** ответ
по найденным фрагментам.
- **GGUF** — формат файла модели для запуска на CPU; **Q4** — квантизация (веса ужаты до
  ~4 бит) → модель весит ~2 ГБ и работает на обычном ноутбуке.
- **llama-cpp** — движок, который запускает GGUF-модели на CPU.
- **Grounded-промпт**: мы жёстко инструктируем модель «отвечай только по фрагментам,
  цитируй номера [n], если ответа нет — верни маркер отказа». Это снижает галлюцинации.
- Работает и на **русском**, и на английском.

### 3.9 Гибридный режим — анти-галлюцинация (`pipeline.py`)
Qwen генерирует ответ, а экстрактивный BERT **проверяет обоснованность**: если BERT тоже
уверенно находит ответ в тех же фрагментах — помечаем «verified»; если нет — «UNVERIFIED
(возможна галлюцинация)». Для банка это важный контроль качества.

## 4. Как мы доказываем качество (метрики)

Нельзя просто сказать «работает» — нужно измерить. Мы считаем:

**Качество поиска (retrieval):**
- **Recall@k** — доля вопросов, у которых нужный фрагмент попал в top-k найденных.
- **MRR** — насколько высоко в списке стоит нужный фрагмент (обратный ранг).

**Качество ответов (end-to-end), официальная метрика SQuAD v2:**
- **Exact Match (EM)** — доля ответов, совпавших с эталоном дословно.
- **F1** — частичное совпадение по словам (мягче EM).
- Отдельно **HasAns** (где ответ есть) и **NoAns** (где ответа нет — качество отказа).

**Честность оценки:**
- **calibration/test split** — порог калибруем на одной части, отчитываемся на другой.
- **oracle-baseline** — прогон ридера на «золотом» контексте; показывает потолок и
  **измеряет потерю на этапе поиска** (у нас ≈ 5 пунктов F1).

**Для генерации (в планах, Pool D):**
- **faithfulness / groundedness** — подтверждён ли ответ найденным контекстом (анти-галлюцинация);
- **латентность** на CPU (генерация медленнее экстракции).

Все числа воспроизводимы: скрипты в `eval/`, результаты (CSV+PNG) в `eval/results/`.

## 5. Приватность и on-premise (почему это козырь для банка)

Все компоненты работают **локально**: эмбеддер, FAISS, BERT и Qwen — на машине банка.
Ни один документ и ни один вопрос **не отправляется во внешние API**. Это снимает риски
утечки, упрощает комплаенс и работает даже без интернета. Для регионального банка это часто
решающий аргумент по сравнению с облачными LLM-сервисами.

## 6. Глоссарий

- **RAG** — поиск релевантных фрагментов + генерация ответа по ним.
- **Чанк (chunk)** — фрагмент документа после нарезки.
- **Токен** — единица текста для модели (≈ часть слова).
- **Эмбеддинг** — числовой вектор, представляющий смысл текста.
- **Косинусная близость** — мера похожести двух векторов.
- **FAISS** — библиотека поиска ближайших векторов.
- **top-k** — сколько самых релевантных фрагментов берём.
- **Экстрактивный QA** — ответ = дословный кусок из текста (span).
- **Генеративный QA** — ответ пишется моделью своими словами по контексту.
- **null-score** — оценка «ответа нет» у экстрактивной модели.
- **τ (tau)** — порог принятия решения «нет ответа».
- **EM / F1** — метрики точности ответа (дословно / по словам).
- **Recall@k / MRR** — метрики качества поиска.
- **Oracle-контекст** — «идеальный» контекст (эталонный) для замера потолка.
- **GGUF / Q4** — формат и квантизация локальной LLM для CPU.
- **llama-cpp** — движок запуска GGUF-моделей.
- **Faithfulness** — обоснованность ответа найденным контекстом.
- **Галлюцинация** — выдуманный, не подтверждённый контекстом ответ.
