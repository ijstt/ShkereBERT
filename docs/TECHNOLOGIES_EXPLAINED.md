## 1. RAG-пайплайн: общая идея

**Вопрос интервьюера:** *Что такое RAG и зачем он нужен?*

**RAG** (Retrieval-Augmented Generation) — архитектура, где модель не отвечает «из головы»,
а сначала **ищет** релевантные фрагменты документа, а потом **читает** ответ из них.

Зачем это нужно:
- LLM (большие языковые модели) могут галлюцинировать — выдумывать факты. RAG привязывает
  ответ к конкретному источнику.
- Документы могут меняться (новые правила банка, новые статьи). Модель не нужно
  переобучать — достаточно обновить корпус.
- Прозрачность: мы можем показать пользователю фрагмент, из которого взят ответ.

Наш пайплайн:
```
Документы → чанкинг → эмбеддинги → FAISS-индекс → retrieval top-k → ридер → ответ + цитата
```

## 2. Чанкинг (разбиение на фрагменты)

**Вопрос интервьюера:** *Зачем вообще резать документ на части? Почему нельзя искать по целому?*

Две причины:
1. **Ридер (BERT) видит ограниченное окно** — максимум 384 токена. Если документ больше,
   ридер просто не увидит часть текста.
2. **Эмбеддинг короткого фрагмента точнее** передаёт его смысл. Эмбеддинг целого
   документа «размазывает» все темы в один вектор — поиск теряет точность.

### Как мы режем

**Sentence-aware стратегия** (по умолчанию):
- Предложения упаковываются жадно в чанки размером ~`size` токенов (по умолчанию 256).
- Если одно предложение длиннее `size`, оно разрезается по токенам (fallback).
- Между соседними чанками есть **overlap** (64 токена) — «перекрытие» не даёт ответу
  разрезаться на границе двух чанков.

**Почему размер в токенах, а не символах?** Потому что модель видит именно токены.
Слово «internationalization» — 1 слово, но несколько токенов. Считая символы, мы бы
неточно оценивали, сколько «увидит» модель.

**Почему 256?** Ablation-исследование (`eval/ablations.py`) показало:
- chunk=128: Recall@1 = 0.854, лучший F1 = 78.1 — лучший recall, но больше чанков.
- chunk=256: Recall@1 = 0.829, F1 = 77.5 — баланс качество/память.
- chunk=512: Recall@1 = 0.824, F1 = 77.3 — крупные чанки начинают «шуметь».

Разница в F1 мала (1 пункт), но на длинных документах (тест `eval_longdoc.py`)
chunk=64 выиграл 5 пунктов F1 у chunk=256: чем меньше постороннего текста видит
ридер, тем чище его решение об абстенции.

## 3. Эмбеддинги и Sentence-Transformers

**Вопрос интервьюера:** *Как вы превращаете текст в вектор? Почему выбрали именно эту модель?*

**Эмбеддинг** — числовой вектор фиксированной длины, который передаёт семантику текста.
Близкие по смыслу тексты получают близкие векторы.

Мы используем **`all-MiniLM-L6-v2`** из библиотеки Sentence-Transformers. Почему:
- Это bi-encoder: вопрос и документ кодируются **независимо**. Это значит, что эмбеддинги
  документов можно посчитать один раз заранее (offline), а при запросе кодировать только
  вопрос — это быстро.
- Модель компактная (22M параметров, 384-мерные векторы), работает на CPU за миллисекунды.
- Обучена на задачах семантического сходства — подходит для поиска.

Для русского языка используем **`paraphrase-multilingual-MiniLM-L12-v2`** — та же архитектура,
но обучена на 50+ языках.

**L2-нормализация**: мы нормализуем все векторы на длину 1. После этого inner product
(скалярное произведение) двух векторов = cosine similarity:

```
cos(a, b) = (a · b) / (|a| · |b|)
```

Если |a| = |b| = 1, то `cos(a, b) = a · b`. Это позволяет использовать быстрый
IndexFlatIP в FAISS (inner product) вместо более дорогого IndexFlatL2.

## 4. FAISS: векторный индекс

**Вопрос интервьюера:** *Что такое FAISS и зачем он нужен? Почему не просто цикл по всем векторам?*

**FAISS** (Facebook AI Similarity Search) — библиотека для быстрого поиска ближайших
соседей в пространстве векторов.

Мы используем `IndexFlatIP` — **точный** (не приближённый) поиск по inner product.
Он перебирает все вектора, но делает это на C++/BLAS с SIMD-оптимизациями, поэтому
работает на порядки быстрее наивного Python-цикла.

**Почему точный, а не приближённый (IVF, HNSW)?** Наш корпус SQuAD — это тысячи, а не
миллионы документов. На таком масштабе точный поиск работает за миллисекунды. Приближённые
индексы дали бы ~0% ускорения, но добавили бы шанс пропустить нужный фрагмент.

**Сохранение на диск**: индекс и чанки сериализуются (`faiss.write_index` + `pickle`),
чтобы не пересобирать при каждом запуске.

## 5. Retrieval: dense-поиск, BM25-гибрид и reranker

**Вопрос интервьюера:** *Как работает поиск? Чем dense-поиск отличается от BM25?*

### Dense (плотный) поиск

1. Вопрос кодируется тем же эмбеддером в вектор.
2. FAISS ищет top-k ближайших векторов чанков.
3. Результат — список `RetrievedChunk` с `dense_score` (cosine similarity).

### BM25

**BM25** — классический словарный (sparse) поиск. Он считает, насколько слова запроса
часто встречаются в документе и насколько они редки во всём корпусе:

```
BM25(q, d) = сумма по словам q: IDF(word) * TF(word, d) * (k1 + 1) / (TF + k1 * (1 - b + b * |d|/avgdl))
```

- **TF** (term frequency) — частота слова в документе.
- **IDF** (inverse document frequency) — обратная частота слова в корпусе; редкие слова
  весят больше.
- **k1, b** — гиперпараметры (по умолчанию ~1.5 и 0.75).

**Зачем нужен BM25 рядом с dense?** Dense-поиск хорошо ловит перефразировки и синонимы,
но может «проплыть» мимо точных терминов, дат, номеров. BM25 ловит их точно (словарное
совпадение).

### Reciprocal Rank Fusion (RRF)

Проблема: скоры BM25 и dense — в разных шкалах (cosine от 0 до 1, BM25 — произвольное
число). Складывать их напрямую бессмысленно.

RRF решает это элегантно — фьюзит **ранги**, а не скоры:

```
RRF(chunk) = 1/(k + rank_dense + 1) + 1/(k + rank_bm25 + 1)
```

где `k` — константа (мы используем 60). Чанк, который высоко в обоих списках, получает
высокий RRF-скор. Подробнее — раздел 19.

**Результат на наших данных**: BM25-гибрид поднял Recall@5 с 0.955 до 0.972 и MRR
с 0.871 до 0.890 за +0.9 мс/запрос. Но end-to-end F1 не изменился (80.11), потому что
вопросы SQuAD — перефразировки (dense и так справляется). Ожидаемая зона выигрыша —
доменные документы с точными терминами.

## 6. Экстрактивный ридер (BERT)

**Вопрос интервьюера:** *Как BERT «находит» ответ в тексте? Что такое span extraction?*

### Суть

Модель `deepset/tinyroberta-squad2` — это RoBERTa (вариант BERT), дообученная на SQuAD v2.
Она получает пару (вопрос, фрагмент) и предсказывает **начало и конец ответа** в тексте
фрагмента (span extraction).

### Как это работает (упрощённо)

1. Вопрос и фрагмент склеиваются: `[CLS] вопрос [SEP] фрагмент [SEP]`.
2. BERT читает всю последовательность и для каждого токена выдаёт два числа:
   - `start_logit[i]` — «насколько этот токен похож на начало ответа».
   - `end_logit[j]` — «насколько этот токен похож на конец ответа».
3. Лучший span: `best_span_score = max(start_logit[i] + end_logit[j])` при `i <= j`
   и `j - i + 1 <= max_answer_len` (30 токенов).

### Обработка «нет ответа» (null-score)

SQuAD v2 содержит вопросы без ответа. Модель обучена предсказывать `[CLS]` как «пустой
ответ»:

```
null_score = start_logit[CLS] + end_logit[CLS]
gap = null_score - best_span_score
```

Если `gap > tau` (порог), модель считает, что ответа нет. Порог `tau` калибруется
отдельно (раздел 13).

### Почему tinyroberta-squad2?

- Маленькая (~80M параметров), быстрая на CPU (~50 мс/вопрос).
- Обучена на SQuAD v2 с поддержкой «нет ответа» из коробки.
- Dev F1 на SQuAD v2 ≈ 81 — это потолок для нашего reader'а (проверено oracle-baseline'ом).

## 7. Генеративный ридер (Qwen)

**Вопрос интервьюера:** *Зачем нужен генеративный ридер, если экстрактивный уже работает?*

### Ограничения экстрактивного ридера

Экстрактивный reader может вернуть только **непрерывный фрагмент текста** из одного чанка.
Он не может:
- Собрать ответ из нескольких фрагментов: «Минимальный взнос 500 руб. Ставка от 14.9%.»
- Перефразировать или обобщить.
- Работать с русским текстом (tinyroberta обучена на английском).

### Решение: Qwen 2.5 3B Instruct

Мы подключили **генеративную LLM** — Qwen 2.5 3B Instruct в формате GGUF через llama-cpp.
Она получает найденные фрагменты и генерирует ответ в свободной форме.

**Grounded generation**: промпт строго ограничивает модель:
1. Отвечать ТОЛЬКО по фрагментам, не использовать внешние знания.
2. Цитировать номера фрагментов: `«Столица — Париж [3].»`
3. Если ответа нет — вернуть маркер `НЕТ ОТВЕТА`.

### Почему офлайн?

Партнёр проекта — банк. Документы не должны покидать периметр организации. Qwen GGUF
на CPU работает полностью локально — никаких API-запросов в облако.

### Метрики

Генеративный ридер даёт строгий F1 = 43.8 (ниже экстрактивного 76.0). Но это не
значит, что он отвечает хуже: метрика F1 (token-level) штрафует за многословие.
Показатель **gold-containment = 0.89** (доля ответов, содержащих правильный текст) почти
равен экстрактивному (0.93). Проблема — Qwen добавляет слова вокруг ответа, и строгий F1
это штрафует.

## 8. Hybrid-режим и анти-галлюцинация

**Вопрос интервьюера:** *Как бороться с галлюцинациями LLM?*

Hybrid-режим — это **Qwen + BERT-верификатор**:
1. Qwen генерирует ответ (естественный, многофактовый, с цитатами).
2. Экстрактивный BERT проверяет, есть ли подтверждение этого ответа в тех же фрагментах.
3. Если BERT не нашёл подтверждения (его `gap > tau`), ответ помечается:
   `UNVERIFIED (возможна галлюцинация)`.

**Результат**: BERT-верификатор перехватывает ~70% галлюцинаций Qwen. Цена — ~0.1 с
дополнительной латентности (один прогон BERT поверх Qwen).

**Пример галлюцинации**: на вопрос «Кто основал Telenet?» при отсутствии ответа в
документе Qwen отвечал из собственной памяти. BERT не находил подтверждения в
фрагменте → пометка UNVERIFIED.

## 9. Абстенция: обработка «нет ответа»

**Вопрос интервьюера:** *Как система понимает, что ответа нет?*

Двухуровневая абстенция:

### Уровень 1: retrieval-level

Если лучший найденный фрагмент имеет `dense_score < min_score` (по умолчанию 0.15),
значит вопрос вообще не похож ни на один документ в базе. Ответ: «нет ответа».

### Уровень 2: reader-level

Фрагмент найден, но reader считает, что в нём нет ответа:
- **Экстрактивный**: `gap = null_score - best_span_score > tau` → нет ответа.
- **Генеративный**: модель возвращает маркер отказа (например, `НЕТ ОТВЕТА`).

### Почему два уровня?

Уровень 1 — быстрый фильтр: если вопрос про погоду, а документ про банковские продукты,
зачем вообще запускать reader? Уровень 2 — тонкий: фрагмент похож на вопрос, но ответа в
нём нет (типичный adversarial-случай SQuAD v2).

## 10. Метрика F1 (token-level)

**Вопрос интервьюера:** *Что такое F1 в контексте QA? Как она считается?*

### Определение

F1 в QA — это **token-level F1**: сравнение предсказанного ответа с эталонным на уровне
отдельных слов (токенов).

### Формула

1. Разбиваем предсказание и эталон на множества слов (после нормализации: lowercase,
   удаление артиклей и знаков препинания).
2. Считаем:
   - **Precision** = |совпавшие токены| / |токены предсказания|
   - **Recall** = |совпавшие токены| / |токены эталона|
   - **F1** = 2 * Precision * Recall / (Precision + Recall)

### Пример

```
Эталон:     "the Battle of Waterloo"  → {"battle", "waterloo"}  (2 токена после нормализации)
Предсказание: "Battle of Waterloo in 1815"  → {"battle", "waterloo", "1815"}  (3 токена)

Совпадения: {"battle", "waterloo"} = 2

Precision = 2/3 = 0.667
Recall    = 2/2 = 1.000
F1        = 2 * 0.667 * 1.000 / (0.667 + 1.000) = 0.800
```

### Почему F1, а не accuracy?

- **Accuracy** (exact match) слишком строга: «Battle of Waterloo» vs «the Battle of
  Waterloo in 1815» — уже неточное совпадение, хотя ответ правильный.
- **F1 прощает частичные совпадения**: лишнее слово или пропущенный артикль не обнуляет
  оценку.

### Наши значения

| setup | F1 | Что это значит |
|---|---|---|
| retrieval-пайплайн | 75.8 | 75.8% token-overlap с эталоном (в среднем по вопросам) |
| oracle-контекст | 81.0 | потолок reader'а (без потерь на retrieval) |

Разница 81.0 − 75.8 = 5.2 F1 — это **потеря на этапе поиска**: retriever не всегда
находит нужный фрагмент.

### Особые случаи

- **Вопрос без ответа (unanswerable)**: если и предсказание, и эталон пусты — F1 = 1.0
  (правильно отказался). Если предсказание непустое, а эталон пуст — F1 = 0.0
  (ложный ответ).
- **Несколько эталонных ответов**: F1 считается для каждого и берётся максимум.
- **HasAns F1 / NoAns F1**: F1 отдельно по answerable и unanswerable вопросам. У нас
  HasAns F1 = 66.2, NoAns F1 = 84.6. NoAns выше, потому что модель хорошо научилась
  отказываться.

## 11. Метрика Exact Match (EM)

**Вопрос интервьюера:** *Чем EM отличается от F1?*

**Exact Match** = доля вопросов, на которые предсказание **точно совпадает** с эталоном
(после нормализации текста). Бинарная метрика: 1 или 0, без частичного кредита.

```
EM("Battle of Waterloo", "the Battle of Waterloo") = 1  (после удаления артикля "the")
EM("Battle of Waterloo", "Battle of Waterloo in 1815") = 0  (лишнее "in 1815")
```

Наш EM = 73.6 (а F1 = 75.8). Разница мала: экстрактивный ридер обычно возвращает точный
span, не добавляя лишнего. У генеративного ридера разрыв EM/F1 больше — он добавляет
слова.

## 12. Recall@k и MRR

**Вопрос интервьюера:** *Как вы оцениваете качество поиска отдельно от ридера?*

### Recall@k

**Recall@k** — доля вопросов, для которых **правильный фрагмент** оказался среди
top-k найденных:

```
Recall@k = (кол-во вопросов, где gold-контекст в top-k) / (всего answerable-вопросов)
```

Считается только по answerable-вопросам (для unanswerable нет «правильного» фрагмента).

Наши значения:

| k | Recall@k | Интерпретация |
|---|---|---|
| 1 | 0.788 | В 79% случаев нужный фрагмент — самый первый |
| 3 | 0.920 | 92% — в тройке лучших |
| 5 | 0.942 | 94% — в пятёрке (наш дефолт top_k=5) |
| 10 | 0.971 | 97% — почти всегда в десятке |

**Почему k=5?** Recall@5 = 94% — уже близко к насыщению (Recall@10 лишь на 3% выше),
а reader обрабатывает каждый фрагмент отдельно, поэтому латентность растёт линейно
с k. При k=1 ~50 мс, k=5 ~250 мс, k=10 ~550 мс. k=5 — оптимум по кривой
качество/скорость.

### MRR (Mean Reciprocal Rank)

**MRR** — средний обратный ранг первого правильного фрагмента:

```
MRR = (1/N) * сумма(1 / rank_i)
```

где `rank_i` — позиция (1-based) первого gold-фрагмента для i-го вопроса.

Пример: если для 3 вопросов gold-чанк оказался на 1-й, 3-й и 2-й позициях:
```
MRR = (1/3) * (1/1 + 1/3 + 1/2) = (1/3) * 1.833 = 0.611
```

Наш MRR = 0.859 — в среднем правильный фрагмент оказывается между 1-й и 2-й позицией.

## 13. Калибровка порога абстенции (tau)

**Вопрос интервьюера:** *Как вы подбираете порог «нет ответа»? Почему tau = −6.3?*

### Проблема

Reader выдаёт `gap = null_score - best_span_score`. Большой gap → модель думает,
что ответа нет. Но какой gap считать «большим»?

### Методология

1. Делим вопросы на **calibration** (50%) и **test** (50%).
2. На calibration прогоняем пайплайн и собираем `gap` для каждого вопроса.
3. Перебираем сетку tau от 2-го до 98-го перцентиля gap'ов (31 значение).
4. Для каждого tau считаем F1 (overall, HasAns, NoAns) по **официальной метрике SQuAD v2**.
5. Выбираем `tau* = argmax F1`.
6. Отчитываемся на **test**-сплите с фиксированным tau* — без утечки.

### Почему это важно

В ранней версии проекта tau подбирался и отчитывался на **одной и той же** выборке.
Это **утечка**: оптимизируешь и оцениваешь на одних данных. F1 был 78.3 — завышен
на ~2.5 пункта. После разделения (calibration/test) честный F1 = 75.8.

### Важная находка

tau* зависит от конфигурации:
- chunk=64: tau* ≈ −9.6
- chunk=512: tau* ≈ −1.7

И от случайной выборки (разные seed'ы): tau* от −9.3 до −4.4.

**Вывод**: порог абстенции **не переносится** между конфигурациями или доменами.
Калибровка обязана быть частью пайплайна. Нельзя один раз подобрать tau и использовать
его везде.

### График

Файл `eval/results/e2e_threshold_curve.png` — кривая F1(tau). Видно, что:
- При слишком низком tau: модель почти не отказывается → высокий HasAns F1, но низкий NoAns F1.
- При слишком высоком tau: модель отказывается слишком часто → наоборот.
- Оптимум — баланс в точке tau*.

## 14. Ablation study

**Вопрос интервьюера:** *Что такое ablation и зачем вы его делали?*

**Ablation study** — систематический эксперимент: меняем ОДИН параметр, фиксируя остальные,
и измеряем влияние на метрики. Это доказательство того, что выбор параметра обоснован,
а не случаен.

### Что мы варьировали

1. **chunk_size** (128, 256, 512): влияние на Recall и F1.
2. **top_k** (1, 3, 5, 10): покрытие vs шум.
3. **reranker on/off**: вклад кросс-энкодера.

### Ключевые выводы

- **top_k**: F1 насыщается к k=3–5, а латентность растёт линейно. При k=10 F1 не
  растёт — дополнительные фрагменты добавляют шум ридеру.
- **chunk_size**: на SQuAD-контекстах (медиана 141 токен) разница минимальна, потому что
  чанкинг почти не срабатывает (при chunk≥256 режется 8% документов). На длинных документах
  разница 5 пунктов F1 — значимая.
- **reranker**: улучшает ранжирование (MRR), но на перефразированных вопросах SQuAD
  прирост F1 невелик. Ожидаемая зона — доменные документы.

Результаты: `eval/results/ablations.csv` и графики `ablation_f1_topk.png`,
`ablation_recall_topk.png`.

## 15. Oracle-baseline: зачем и что он показывает

**Вопрос интервьюера:** *Что такое oracle-baseline и зачем он нужен?*

**Oracle-baseline** — это прогон reader'а на **золотом (правильном) контексте** вопроса,
без retrieval. Он отвечает на вопрос: «Какой максимальный F1 мог бы быть, если бы
retriever всегда находил нужный фрагмент?»

### Что он показал

| setup | F1 |
|---|---|
| retrieval-пайплайн | 75.8 |
| oracle-контекст | 81.0 |

Потеря на retrieval: 81.0 − 75.8 = **5.2 пункта F1**. Это количественная мера того,
сколько качества мы теряем из-за несовершенного поиска.

Oracle F1 = 81.0 совпал с эталонным dev-F1 модели tinyroberta-squad2 на SQuAD v2.
Это значит, что наша реализация reader'а **корректна** — мы не теряем качество из-за
багов в коде извлечения span'ов.

## 16. SQuAD v2: почему именно этот датасет

**Вопрос интервьюера:** *Почему SQuAD v2, а не SQuAD v1 или другой датасет?*

**SQuAD v2** = SQuAD v1 + ~50,000 unanswerable-вопросов, составленных так, чтобы быть
похожими на answerable (adversarial). Это делает его идеальным для нашей задачи:

1. **Unanswerable-вопросы**: наша система обязана говорить «ответа нет». SQuAD v1 не
   содержит таких вопросов — на нём нельзя оценить абстенцию.
2. **Стандартизированные метрики**: EM/F1 — общепринятые, есть готовая реализация в
   библиотеке `evaluate`.
3. **Gold-контексты**: для каждого вопроса известен правильный контекст — это позволяет
   строить oracle-baseline и оценивать retrieval отдельно от reader'а.

### Ограничение

SQuAD — энциклопедический (Wikipedia). Контексты короткие (медиана ~141 токен). На
узкоспециальных или длинных документах качество может отличаться. Мы честно это показали
через `eval_longdoc.py` (тест на длинных документах).

## 17. Дисковый кэш эмбеддингов

**Вопрос интервьюера:** *Зачем кэшировать эмбеддинги? Это же быстро считается.*

Нет — на CPU это **самая дорогая операция**. Для 1000 чанков эмбеддирование занимает
~10–30 секунд. При каждом запуске eval-скрипта или ablation-свипа это суммируется.

### Реализация

Ключ кэша — SHA1-хэш от:
- имени модели эмбеддера
- параметров чанкинга (size, overlap, strategy)
- id всех чанков

Если параметры изменились — кэш автоматически пересчитывается. Если нет — загружается
с диска (`numpy.load`).

Это позволяет ablation-свипу по `top_k` (при фиксированном чанкинге) работать **без
перерасчёта эмбеддингов** — экономия десятков минут.

## 18. llama-cpp и GGUF: зачем локальная LLM

**Вопрос интервьюера:** *Зачем использовать llama-cpp, а не API OpenAI/Anthropic?*

### llama-cpp

**llama-cpp** — C/C++ runtime для инференса LLM. Работает на CPU (и опционально GPU).
Python-обёртка — `llama-cpp-python`.

### GGUF

**GGUF** — бинарный формат весов модели с поддержкой квантизации. Квантизация **Q4_K_M**
означает: веса хранятся в 4-битном формате (вместо 16-бит). Модель Qwen 2.5 3B уменьшается
с ~6 GB до ~2 GB, ускоряется на CPU, а качество теряется минимально.

### Почему не API?

1. **Приватность**: документы банка не должны покидать периметр. API = отправка текста
   в облако.
2. **Стоимость**: API — платный за токен. Локальная модель — бесплатно после скачивания.
3. **Доступность**: работает офлайн, не зависит от сети и доступности сервиса.
4. **Детерминированность**: `temperature=0.0` — одинаковый ответ при повторных запросах.

### Компромисс

Qwen 3B на CPU медленнее, чем API (10 с/вопрос vs ~1 с). Но для банковского
on-premise это приемлемо: приватность важнее скорости.

## 19. Reciprocal Rank Fusion (RRF)

**Вопрос интервьюера:** *Как объединить результаты двух разных поисковых систем?*

### Проблема

Dense-поиск возвращает cosine similarity (от 0 до 1). BM25 возвращает скоры на своей
шкале (может быть 0–50). Складывать эти числа напрямую — как складывать метры и килограммы.

### Решение: RRF

Вместо скоров используем **ранги** (позиции в отсортированном списке):

```
RRF(chunk) = 1/(k + rank_dense + 1) + 1/(k + rank_bm25 + 1)
```

- `k = 60` — константа сглаживания (стандартное значение из оригинальной статьи).
- `rank` — 0-based позиция чанка в каждом списке.
- Чанк, не найденный одной из систем, получает вклад только от другой.

### Пример

Чанк A: dense rank=0, bm25 rank=2.
```
RRF(A) = 1/(60+0+1) + 1/(60+2+1) = 1/61 + 1/63 = 0.0164 + 0.0159 = 0.0323
```

Чанк B: dense rank=5, bm25 rank=0.
```
RRF(B) = 1/(60+5+1) + 1/(60+0+1) = 1/66 + 1/61 = 0.0152 + 0.0164 = 0.0316
```

Оба высоко, но A чуть выше, потому что его ранги сбалансированнее.

### Важный нюанс в нашей реализации

`dense_score` (cosine) сохраняется отдельно от RRF-скора. Почему? Retrieval-level
абстенция (`dense_score < min_score`) работает на шкале cosine. Если бы мы использовали
RRF-скор, порог `min_score` потерял бы смысл.

## 20. Cross-encoder (reranker)

**Вопрос интервьюера:** *Чем reranker отличается от bi-encoder? Зачем он нужен?*

### Bi-encoder (наш основной эмбеддер)

Кодирует вопрос и документ **независимо**. Быстро (можно заранее посчитать эмбеддинги
документов), но не видит взаимодействия между словами вопроса и документа.

```
score = cosine(embed(вопрос), embed(документ))
```

### Cross-encoder (reranker)

Кодирует пару (вопрос, документ) **вместе** через один проход Transformer:

```
score = CrossEncoder(вопрос + [SEP] + документ)
```

Видит **все** взаимодействия между словами — более точен, но **медленный**: нельзя
заранее посчитать, нужно прогонять для каждой пары.

### Как мы используем

1. Bi-encoder находит top-20 кандидатов (быстро).
2. Cross-encoder (`ms-marco-MiniLM-L-6-v2`) переоценивает эти 20 кандидатов (точно).
3. Берём top-k (5) из переоценённых.

Это **двухступенчатый retrieval**: грубый фильтр (bi-encoder) + точное ранжирование
(cross-encoder). Стандартный паттерн в production-системах.

## 21. Frozen dataclasses для конфигурации

**Вопрос интервьюера:** *Почему конфигурация через frozen dataclasses, а не словарь/YAML?*

### Frozen dataclasses (`@dataclass(frozen=True)`)

- **Неизменяемость**: после создания объект нельзя изменить. Это гарантирует, что
  конфигурация не мутирует где-то посреди пайплайна (трудноловимые баги).
- **Типизация**: каждый параметр имеет тип и значение по умолчанию. IDE/линтер
  подсказывает доступные параметры.
- **Вложенность**: `Config` содержит `ChunkConfig`, `RetrieverConfig`, `ReaderConfig`,
  `GeneratorConfig` — каждый модуль видит только свою секцию.
- **Переопределение через `replace()`**: из-за `frozen=True` нельзя написать
  `cfg.chunk.size = 128`. Вместо этого: `replace(cfg.chunk, size=128)` — создаёт
  новый объект. Это явно и прозрачно.

Утилита `with_overrides(cfg, chunk__size=128)` делает это через двойное подчёркивание —
удобно для CLI и ablation-скриптов.

## 22. Multi-seed оценка и доверительные интервалы

**Вопрос интервьюера:** *Как вы убеждаетесь, что ваши метрики — не случайность?*

### Проблема

Мы оцениваем на подвыборке из SQuAD (500–1500 вопросов). Разные подвыборки дадут
разные метрики. Разница в 1–2 F1 между конфигурациями может быть **шумом выборки**.

### Решение: multi-seed

`eval/eval_multiseed.py` прогоняет полный протокол (свой корпус → калибровка tau →
отчёт на test) на **5 разных seed'ах** (разные подвыборки вопросов).

Результат: F1 = 76.9 ± 3.9 (95% CI).

**Что это значит**: с 95% вероятностью «истинный» F1 лежит в интервале [73.0, 80.8].
Наш основной результат (75.8) — внутри интервала, то есть он не выброс.

**Практический вывод**: разница конфигураций в 1–2 F1 на малых выборках — шум.
Доверяем только эффектам крупнее CI (~4 F1). Эффект чанкинга на длинных документах
(~5 F1) — значимый.

## 23. Ключевые инженерные решения и находки

**Вопрос интервьюера:** *С какими «граблями» столкнулись и что из этого вынесли?*

### 1. Батчинг ридера на CPU не помогает

Пробовали обрабатывать k фрагментов одним forward-pass BERT. На CPU это **не ускорило**:
паддинг до самого длинного фрагмента добавляет «пустой» compute. Пофрагментная обработка
(каждый фрагмент в нативной длине) оказалась быстрее.

**Вывод**: батчинг ридера выгоден на GPU (параллелизм компенсирует паддинг), но не на CPU.

### 2. sentence-transformers 5.x ломает офлайн-загрузку

Версия 5.x грузит модели через `AutoProcessor` (вместо `AutoTokenizer`), что ломает
`local_files_only` для многоязычных моделей. Решение: пин `sentence-transformers<5`
в requirements.

### 3. Qwen отвечает из «памяти» на unanswerable-вопросы

На adversarial-вопросах без ответа (например, про Telenet) Qwen уверенно отвечал из
своих знаний (выглядит убедительно, но это галлюцинация по отношению к документу).
Решение: явный запрет внешних знаний в промпте + UNVERIFIED-метка в hybrid-режиме.

### 4. Бесконечный цикл в чанкере

Длинное предложение без пунктуации + overlap, равный длине чанка → цикл не продвигался.
Решение: переписали на `for`-цикл с гарантией продвижения (каждое предложение
добавляется ровно один раз).

### 5. Утечка при калибровке tau

Подбирали tau и отчитывались на **одной выборке**. F1 был завышен на ~2.5 пункта.
После разделения на calibration/test — честные 75.8.

**Урок**: метрики нужно всегда считать на данных, которые не участвовали в оптимизации
гиперпараметров (в том числе порогов).

### 6. Русский промпт для английских вопросов

System-prompt на русском провоцировал Qwen отвечать по-русски на английские вопросы.
Решение: промпт выбирается по языку вопроса (детекция кириллицы).

### 7. sentencepiece — неочевидная зависимость

Многоязычный эмбеддер (XLM-R) использует sentencepiece-токенизатор. Без пакета
`sentencepiece` загрузка падает с непонятным `Unrecognized processing class`.
Добавлен в requirements.
