## 1. Что такое нейросеть

**Вопрос интервьюера:** *Объясни простыми словами, что такое нейронная сеть.*

### Аналогия

Представьте сборочную линию на заводе. На вход поступает сырьё (данные — текст, картинка,
число). На каждом этапе (слое) рабочие (нейроны) обрабатывают то, что получили: что-то
усиливают, что-то приглушают. На выходе — готовый продукт (предсказание).

### Как это устроено

Нейросеть — это цепочка **слоёв**. Каждый слой — набор **нейронов**. Нейрон делает
три вещи:

1. **Принимает входы** — числа от предыдущего слоя.
2. **Умножает каждый вход на свой вес** и складывает (линейная комбинация):
   ```
   z = w1*x1 + w2*x2 + ... + wn*xn + b
   ```
   - `w1, w2, ...` — **веса** (параметры, которые модель учит).
   - `b` — **смещение** (bias).
3. **Применяет функцию активации** — нелинейное преобразование (например, ReLU: если
   z < 0, вернуть 0; иначе вернуть z). Без нелинейности вся сеть свелась бы к одному
   линейному преобразованию (перемножение матриц), и не смогла бы выучить сложные
   зависимости.

### Обучение (кратко)

Модель предсказывает ответ → сравнивает его с правильным → считает **ошибку** (loss) →
корректирует веса так, чтобы ошибка уменьшилась (градиентный спуск + обратное
распространение ошибки). Это повторяется миллионы раз.

### Применительно к проекту

- `tinyroberta-squad2` — нейросеть с ~80 миллионами весов.
- `Qwen 2.5 3B` — нейросеть с ~3 миллиардами весов.
- Мы **не обучаем** эти сети (это уже сделано авторами). Мы **используем** их
  обученные веса для предсказаний (инференс).

## 2. Что такое Transformer и механизм внимания

**Вопрос интервьюера:** *Все говорят «Transformer». Что это и почему это важно?*

### До Transformer

Раньше для обработки текста использовали **рекуррентные** сети (RNN, LSTM). Они читали
текст **последовательно** — слово за словом, как человек. Проблемы:
- Медленно: нельзя параллелить (каждое слово зависит от предыдущего).
- Забывает начало длинного текста (информация «затухает»).

### Transformer (2017, статья «Attention Is All You Need»)

Transformer читает **все слова одновременно** и для каждого слова решает, на какие
другие слова обратить внимание. Это и есть **механизм внимания** (attention).

### Механизм внимания — интуиция

Представьте, что вы читаете предложение:

> «Банк выдал кредит на покупку квартиры в 2023 году.»

Для слова «кредит» важны слова «банк» и «покупку» — они дают контекст. Слово «году»
менее важно. Attention позволяет модели **автоматически** вычислить, какие слова важны
для каждого другого слова.

### Как это работает (упрощённо)

Для каждого слова создаются три вектора:
- **Query** (Q) — «что я ищу?»
- **Key** (K) — «что я могу предложить?»
- **Value** (V) — «что я содержу?»

Скор внимания между словами i и j:
```
attention(i, j) = softmax(Q_i · K_j / sqrt(d))
```
Чем больше скор, тем сильнее слово j влияет на представление слова i. Итоговое
представление — взвешенная сумма Value-векторов.

### Почему это прорыв

1. **Параллелизм**: все слова обрабатываются одновременно → GPU может считать
   быстро.
2. **Дальние связи**: слово в начале предложения «видит» слово в конце без потерь.
3. **Масштабируемость**: Transformer можно увеличивать (больше слоёв, больше данных)
   и он продолжает улучшаться. Так появились GPT, BERT, Qwen и другие.

### В нашем проекте

- **BERT** (экстрактивный ридер) — Transformer-encoder (читает текст целиком).
- **Qwen** (генеративный ридер) — Transformer-decoder (генерирует текст слово за словом).
- **Sentence-Transformers** (эмбеддер) — Transformer-encoder, выдающий один вектор на
  весь текст.

## 3. Что такое BERT

**Вопрос интервьюера:** *Что такое BERT и почему он «понимает» текст?*

### Расшифровка

**BERT** = Bidirectional Encoder Representations from Transformers.

- **Bidirectional** — читает текст в обе стороны одновременно (а не слева направо,
  как GPT).
- **Encoder** — часть Transformer, которая превращает текст в внутреннее представление.
- **Representations** — модель создаёт числовое представление каждого токена,
  учитывающее контекст.

### Как BERT учится «понимать» язык: предобучение

BERT обучен на огромном корпусе текстов (Wikipedia + BookCorpus, ~3 млрд слов) двумя
задачами:

**1. Masked Language Modeling (MLM):**
- Случайно прячем 15% слов: `«Кот [MASK] на [MASK]»`.
- Модель должна угадать: `«Кот сидел на крыше»`.
- Так она учит грамматику, семантику, связи между словами.

**2. Next Sentence Prediction (NSP):**
- Даём два предложения. Модель решает, идёт ли второе после первого в оригинале.
- Так она учит связи между предложениями.

### Файн-тюнинг (дообучение)

После предобучения BERT «понимает» язык, но не умеет отвечать на вопросы. Файн-тюнинг —
это дообучение на конкретной задаче:

- Берём предобученный BERT.
- Добавляем «голову» — два линейных слоя (start-слой и end-слой).
- Обучаем на SQuAD: модель учится предсказывать, где в тексте начинается и заканчивается
  ответ.

Результат: `tinyroberta-squad2` — предобученная RoBERTa (улучшенный BERT) +
файн-тюнинг на SQuAD v2.

### Почему это работает

Предобучение на миллиардах слов научило модель «понимать» язык на глубоком уровне:
синтаксис, семантику, контекст. Файн-тюнинг лишь «настраивает» это понимание на
конкретную задачу. Без предобучения модели пришлось бы учить язык с нуля на
маленьком датасете — это невозможно.

## 4. Что такое токенизация

**Вопрос интервьюера:** *Почему модель работает с «токенами», а не со словами?*

### Проблема с целыми словами

Если каждое слово — отдельный элемент словаря:
- Словарь огромен (миллионы слов с формами, сленгом, ошибками).
- Новые/редкие слова → `<UNK>` (неизвестный токен), модель не может с ними работать.

### Решение: subword-токенизация

Разбиваем слова на **подслова** (subwords):
```
"internationalization" → ["inter", "##nation", "##alization"]
"котёнок"             → ["кот", "##ёнок"]
```

Преимущества:
- Словарь компактный (~30,000 токенов).
- Редкие слова разбиваются на знакомые части — модель всё равно «понимает» их.
- Частые слова остаются целыми: `«the»`, `«кот»`.

### Виды токенизаторов

| Метод | Кто использует | Суть |
|---|---|---|
| **WordPiece** | BERT, RoBERTa | Жадно разбивает слово, максимизируя правдоподобие |
| **BPE** (Byte Pair Encoding) | GPT, Qwen | Итеративно сливает самые частые пары символов |
| **SentencePiece** | XLM-R, multilingual модели | BPE/Unigram, работает на «сыром» тексте (без предварительной токенизации по пробелам — подходит для языков без пробелов: китайский, японский) |

### Пример BPE

Корпус: `«low lower lowest lowly»`.

1. Начинаем с символов: `l, o, w, e, r, s, t, l, y, ...`
2. Самая частая пара: `l + o` → сливаем в `lo`.
3. Следующая: `lo + w` → `low`.
4. И так далее, пока не наберём нужный размер словаря.

### В нашем проекте

Мы считаем размер чанка **в токенах** (а не в словах/символах), потому что модель
видит именно токены. Слово «internationalization» — 1 слово, но может быть 4–5 токенов.
Если считать символами, мы неверно оценим, сколько «влезет» в окно модели.

`TokenCounter` в `chunking.py` использует токенизатор эмбеддера для точного подсчёта.

## 5. Что такое transfer learning

**Вопрос интервьюера:** *Почему вы не обучили свою модель с нуля? Что такое
transfer learning?*

### Аналогия

Представьте, что вам нужно научиться играть в теннис. Два пути:
1. Начать с нуля: учиться держать ракетку, бегать, координировать движения.
2. Вы уже спортсмен (бадминтон): рефлексы, координация, стратегия — уже есть.
   Нужно лишь адаптироваться под теннис.

Transfer learning — это путь 2.

### Как это работает в ML

1. **Предобучение** (pre-training): модель обучается на огромном корпусе (Wikipedia,
   книги, интернет) общей задаче — «понимать язык». Это дорого: тысячи GPU, недели
   вычислений, миллионы долларов.
2. **Дообучение** (fine-tuning): берём предобученную модель и обучаем на маленьком
   датасете конкретной задаче (ответы на вопросы, классификация, перевод). Это дёшево:
   один GPU, часы.

### Почему это важно

- **Эффективность**: предобучение сделано один раз (авторами BERT, Qwen и т.д.).
  Мы переиспользуем результат.
- **Качество**: модель, «знающая» язык, выучит новую задачу лучше и быстрее, чем
  модель с нуля.
- **Данные**: для обучения с нуля нужны миллионы примеров. Для fine-tuning — тысячи.

### В нашем проекте

Мы используем **три** предобученные модели:

| Модель | Предобучение | Fine-tuning | Наше использование |
|---|---|---|---|
| `all-MiniLM-L6-v2` | Общий корпус (семантическое сходство) | — | Эмбеддинги (as-is) |
| `tinyroberta-squad2` | RoBERTa (MLM на 160GB текста) | SQuAD v2 | Экстрактивный QA |
| `Qwen 2.5 3B` | Огромный корпус (триллионы токенов) | Instruction-tuning | Генеративный QA |

Мы **не дообучаем** ни одну из них — используем как есть (zero-shot / out-of-the-box).
Это осознанный выбор: у нас нет GPU и размеченных данных для дообучения, а качество
«из коробки» достаточно для задачи.

## 6. Что такое логиты, softmax, вероятность

**Вопрос интервьюера:** *Что модель реально выдаёт? Что такое логиты?*

### Логиты

Нейросеть на последнем слое выдаёт **сырые числа** — логиты. Они могут быть любыми:
положительными, отрицательными, большими, маленькими. Сами по себе они не вероятности
(не от 0 до 1, не суммируются в 1).

Пример: модель предсказывает следующее слово. Логиты для трёх вариантов:
```
"кот":    5.2
"собака": 3.1
"рыба":   0.4
```
«Кот» — самый вероятный, но 5.2 — это не вероятность.

### Softmax

Чтобы превратить логиты в вероятности, применяем **softmax**:

```
P(класс_i) = exp(logit_i) / сумма(exp(logit_j) для всех j)
```

Для нашего примера:
```
exp(5.2) = 181.3
exp(3.1) = 22.2
exp(0.4) = 1.5

P("кот")    = 181.3 / (181.3 + 22.2 + 1.5) = 0.884  (88.4%)
P("собака") = 22.2 / 205.0                  = 0.108  (10.8%)
P("рыба")   = 1.5 / 205.0                   = 0.007  (0.7%)
```

Теперь сумма = 1.0, каждое число от 0 до 1 — вероятности.

### Свойства softmax

- **Экспонента** усиливает различия: маленькая разница в логитах → большая разница
  в вероятностях.
- **Нормализация** (делим на сумму) гарантирует, что сумма = 1.
- Softmax **не меняет порядок**: если логит A > логит B, то P(A) > P(B).

### В нашем проекте

Экстрактивный ридер выдаёт `start_logits` и `end_logits` — логиты для каждого
токена. Мы **не** применяем softmax, а работаем с сырыми логитами напрямую:
- `best_span_score = start_logit[i] + end_logit[j]` — сумма логитов лучшего span.
- `null_score = start_logit[CLS] + end_logit[CLS]` — логит «пустого ответа».
- `gap = null_score - best_span_score` — разница определяет абстенцию.

Softmax не нужен: нам важна **разница** между скорами, а не абсолютные вероятности.
Softmax сохраняет порядок, но скрывает масштаб разницы.

## 7. Что такое косинусное сходство

**Вопрос интервьюера:** *Как измерить «похожесть» двух текстов числом?*

### Интуиция

Каждый текст — это вектор (стрелка) в многомерном пространстве. **Косинусное сходство**
измеряет **угол** между двумя стрелками:
- Угол 0° (параллельные, одного направления) → cosine = 1.0 → тексты очень похожи.
- Угол 90° (перпендикулярные) → cosine = 0.0 → тексты не связаны.
- Угол 180° (противоположные) → cosine = −1.0 → тексты противоположны по смыслу.

### Формула

```
cosine(A, B) = (A · B) / (|A| * |B|)
```

- `A · B` — скалярное произведение (поэлементно умножить и сложить).
- `|A|` — длина вектора (корень из суммы квадратов).

### Пример (в 2D для наглядности)

Вектор «кот» = (3, 4), вектор «котёнок» = (6, 8), вектор «самолёт» = (−2, 5).

```
cosine(кот, котёнок) = (3*6 + 4*8) / (5 * 10) = 50/50 = 1.0   ← идентичное направление!
cosine(кот, самолёт) = (3*(-2) + 4*5) / (5 * 5.39) = 14/26.9 = 0.52  ← частично похожи
```

«Кот» и «котёнок» указывают в одном направлении (только «котёнок» длиннее — но
косинус игнорирует длину, только угол).

### Почему косинус, а не евклидово расстояние?

**Евклидово расстояние** (`sqrt((x1-x2)^2 + (y1-y2)^2)`) зависит от **длины** вектора.
Два одинаковых по смыслу текста разной длины (короткое и длинное предложение) могут
получить далёкие по евклидовой метрике векторы, но одинаковое направление.

**Косинус** зависит только от направления — это то, что нам нужно для семантического
сходства.

### В нашем проекте

Мы L2-нормализуем все векторы (приводим к длине 1). После этого:
```
cosine(A, B) = A · B    (т.к. |A| = |B| = 1)
```
Это позволяет использовать FAISS `IndexFlatIP` (inner product) вместо более дорогого
косинусного поиска — математически эквивалентно, но быстрее.

## 8. Что такое векторное пространство

**Вопрос интервьюера:** *Что значит «текст = точка в пространстве»?*

### Идея

Мы привыкли к 2D (плоскость) и 3D (пространство). Но математически ничто не мешает
работать с 384-мерным пространством (именно столько мерностей у нашего эмбеддера
MiniLM).

Каждый текст превращается в **вектор из 384 чисел** — точку в 384-мерном пространстве.

### Визуальная аналогия (в 2D)

Представьте карту:
- Все тексты про «спорт» — в правом верхнем углу.
- Все тексты про «финансы» — в левом нижнем.
- Текст «инвестиции в спортивные клубы» — где-то посередине.

В реальности мерностей 384, а не 2, но принцип тот же: **близкие по смыслу тексты
расположены рядом**.

### Почему это работает

Модель-эмбеддер (MiniLM) обучена так, чтобы:
- Похожие тексты → близкие векторы (маленький угол).
- Разные тексты → далёкие векторы (большой угол).

Это не «магия» — это результат обучения на миллионах пар текстов, где модель
подбирала веса, чтобы пары «вопрос-ответ», «заголовок-статья» были близко, а
случайные пары — далеко.

### В нашем проекте

1. Все чанки документов превращаются в точки 384-мерного пространства.
2. Вопрос пользователя тоже превращается в точку.
3. FAISS ищет ближайших соседей вопроса среди чанков → это и есть «поиск по смыслу».

## 9. Precision и Recall

**Вопрос интервьюера:** *Что такое precision и recall? Чем они отличаются?*

### Простая аналогия

Вы ищете грибы в лесу.

- **Precision** (точность) — «из всех грибов, что я собрал, сколько съедобных?»
  Если собрали 10, а съедобных 8 → precision = 80%.
- **Recall** (полнота) — «из всех съедобных грибов в лесу, сколько я нашёл?»
  Если в лесу 20 съедобных, а нашли 8 → recall = 40%.

### Формулы

```
Precision = TP / (TP + FP)  — «сколько из найденного правильно»
Recall    = TP / (TP + FN)  — «сколько из правильного найдено»
```

(TP, FP, FN — см. раздел 10.)

### Баланс

- **Высокий precision, низкий recall**: «Я нашёл мало, но всё правильное» (осторожная
  система).
- **Высокий recall, низкий precision**: «Я нашёл всё, но набрал кучу мусора»
  (агрессивная система).

F1 = гармоническое среднее precision и recall — баланс между ними:
```
F1 = 2 * P * R / (P + R)
```

### В нашем проекте (token-level)

В QA precision и recall считаются **по токенам ответа**:
- **Precision**: «из слов, что модель выдала, сколько есть в эталоне?»
- **Recall**: «из слов эталона, сколько модель выдала?»

Пример:
```
Эталон:      {"battle", "waterloo"}
Предсказание: {"battle", "waterloo", "1815"}

Precision = 2/3 = 0.667   (лишнее слово "1815")
Recall    = 2/2 = 1.000   (все слова эталона найдены)
F1        = 2 * 0.667 * 1.000 / 1.667 = 0.800
```

## 10. Confusion matrix: TP, FP, FN, TN

**Вопрос интервьюера:** *Что такое confusion matrix и зачем она нужна?*

### Определение

Confusion matrix — таблица, которая показывает, сколько предсказаний модели попали
в каждую из 4 категорий:

|  | Реально положительный | Реально отрицательный |
|---|---|---|
| **Модель сказала «да»** | TP (True Positive) — правильно нашёл | FP (False Positive) — ложная тревога |
| **Модель сказала «нет»** | FN (False Negative) — пропустил | TN (True Negative) — правильно отверг |

### Пример из жизни: спам-фильтр

- **TP**: письмо — спам, фильтр пометил как спам. Отлично.
- **FP**: письмо — не спам, фильтр пометил как спам. Плохо (важное письмо потеряно).
- **FN**: письмо — спам, фильтр пропустил. Плохо (спам в ящике).
- **TN**: письмо — не спам, фильтр не тронул. Отлично.

### В нашем проекте: HasAns / NoAns

В контексте SQuAD v2 с абстенцией:

|  | Есть ответ (HasAns) | Нет ответа (NoAns) |
|---|---|---|
| **Модель ответила** | Правильно ответила (TP) | Ложный ответ — галлюцинация (FP) |
| **Модель отказалась** | Пропустила ответ (FN) | Правильно отказалась (TN) |

- **HasAns F1 = 66.2**: как хорошо модель отвечает, когда ответ есть.
- **NoAns F1 = 84.6**: как хорошо модель отказывается, когда ответа нет.

NoAns F1 выше — модель лучше справляется с отказом, чем с точным извлечением ответа.
Это объяснимо: сказать «нет ответа» — бинарное решение; извлечь точный span — сложнее.

## 11. Train / Val / Test split

**Вопрос интервьюера:** *Зачем делить данные на три части? Почему нельзя обучать и
оценивать на одних данных?*

### Аналогия

Учитель составляет тест для школьников.
- Если дать школьникам **те же задачи**, на которых они тренировались, все получат
  5 — но это не значит, что они поняли тему.
- Правильно: тренироваться на одних задачах, сдавать **новые** — это проверяет
  понимание.

### Три части

| Набор | Зачем | Аналогия |
|---|---|---|
| **Train** (обучающий) | Обучение модели | Задачи для тренировки |
| **Validation** (валидационный) | Подбор гиперпараметров | Пробный тест (проверяем подготовку, можно менять стратегию) |
| **Test** (тестовый) | Финальная оценка | Экзамен (смотрим один раз, честная оценка) |

### Утечка данных (data leakage)

Если случайно использовать test-данные при обучении или подборе гиперпараметров —
оценка будет **завышена**. Модель «видела» ответы и «подстроилась» под них.

### В нашем проекте

Мы не обучаем модели, но у нас есть **порог абстенции tau** — это гиперпараметр.
В ранней версии мы подбирали tau и отчитывались на **одной и той же** выборке.
Это утечка:
- F1 был 78.3 (завышен).
- После разделения на calibration (подбор tau) и test (отчёт): F1 = 75.8.
- Разница ~2.5 пункта — это и есть цена утечки.

**Правило**: никогда не оценивай на данных, которые использовал для оптимизации чего-либо.

## 12. Гиперпараметры vs параметры модели

**Вопрос интервьюера:** *В чём разница между параметрами и гиперпараметрами?*

### Параметры модели

**Параметры** — это **веса нейросети**, которые модель **сама выучивает** во время
обучения. У tinyroberta их ~80 миллионов. Мы их не трогаем.

Примеры:
- Веса attention-слоёв BERT.
- Веса линейных слоёв.
- Bias-ы.

### Гиперпараметры

**Гиперпараметры** — это **настройки**, которые задаём **мы** (разработчики) до
запуска. Модель их не учит.

Примеры из нашего проекта:

| Гиперпараметр | Что он делает | Наше значение |
|---|---|---|
| `chunk_size` | Размер чанка в токенах | 256 |
| `overlap` | Перекрытие чанков | 64 |
| `top_k` | Сколько фрагментов искать | 5 |
| `min_score` | Порог retrieval-абстенции | 0.15 |
| `tau` | Порог reader-абстенции | −6.3 |
| `max_answer_len` | Макс. длина ответа | 30 токенов |
| `temperature` | Креативность LLM | 0.0 (детерминированный) |

### Как подбирать гиперпараметры

- **Вручную** (expert knowledge): «256 токенов — разумный размер чанка».
- **Grid search**: перебрать все комбинации (наш ablation study: chunk_size × top_k).
- **На валидации**: подобрать так, чтобы метрика на validation-выборке была максимальной.
  Отчитываться — на test.

## 13. Overfitting и underfitting

**Вопрос интервьюера:** *Что такое переобучение? Как его обнаружить?*

### Аналогия

Студент готовится к экзамену:
- **Overfitting** (переобучение): выучил ответы на конкретные билеты наизусть. На тех
  же билетах — отлично. На новых — не может ответить. «Зазубрил, но не понял».
- **Underfitting** (недообучение): учил мало. И на старых, и на новых билетах — плохо.
  «Не выучил достаточно».
- **Правильное обучение**: понял принципы. На старых — хорошо, на новых — тоже хорошо.

### Как обнаружить

| Ситуация | Train-метрика | Test-метрика | Диагноз |
|---|---|---|---|
| Всё плохо | Низкая | Низкая | Underfitting |
| Подозрительно | Высокая | Низкая | Overfitting |
| Хорошо | Высокая | Высокая | Нормально |

### Как бороться

- **С underfitting**: увеличить модель, обучать дольше, дать больше данных.
- **С overfitting**: больше данных, регуляризация (dropout, weight decay), ранняя
  остановка, аугментация данных.

### В нашем проекте

Мы не обучаем модели, поэтому классический overfitting нам не грозит. Но есть
**аналогичная проблема** — overfitting гиперпараметров:

Если подбираем tau на test-данных и на них же оцениваем — мы «переобучаем» порог
под конкретную выборку. Он будет работать хуже на новых данных. Именно поэтому
мы разделили calibration/test.

Multi-seed оценка (5 seed'ов, F1 = 76.9 ± 3.9) показала, что наш результат
**стабилен** — не случайный выброс на удачной выборке.

## 14. CPU vs GPU инференс

**Вопрос интервьюера:** *Почему ML обычно работает на GPU? Почему мы на CPU?*

### CPU (Central Processing Unit)

- **Мало ядер** (4–32), но каждое мощное.
- Хорош для **последовательных** задач: «сделай одну сложную операцию».
- Есть на каждом компьютере.

### GPU (Graphics Processing Unit)

- **Тысячи ядер**, но каждое слабее.
- Хорош для **параллельных** задач: «сделай 10,000 одинаковых простых операций
  одновременно».
- Нейросети — идеальная задача для GPU: матричные умножения (тысячи одинаковых
  операций параллельно).

### Почему мы на CPU

1. **On-premise** (банк): не на каждом сервере есть GPU. CPU — гарантированно есть.
2. **Стоимость**: GPU-сервер стоит в 5–10 раз дороже.
3. **Наши модели маленькие**:
   - MiniLM (22M параметров) — эмбеддинг за ~1 мс/текст на CPU.
   - tinyroberta (80M) — инференс за ~50 мс/вопрос.
   - Qwen 3B — самая тяжёлая, ~10 с/вопрос на CPU. Это приемлемо для QA-бота.

### Инженерная находка

Мы пробовали батчить reader (обрабатывать k фрагментов одним forward-pass).
На GPU это ускорило бы: все k фрагментов считаются параллельно. На CPU — нет:
паддинг до самого длинного фрагмента добавляет «пустой» compute. Пофрагментная
обработка (каждый в нативной длине, без паддинга) оказалась быстрее на CPU.

## 15. Квантизация моделей

**Вопрос интервьюера:** *Как модель с 3 миллиардами параметров помещается на обычном
компьютере?*

### Проблема

Каждый параметр модели — число. Стандартное хранение:
- **FP32** (32 бита / 4 байта) — полная точность.
- 3B параметров × 4 байта = **12 GB** — не влезает в RAM многих компьютеров.

### Решение: квантизация

**Квантизация** — уменьшение точности хранения чисел:

| Формат | Бит на параметр | Размер 3B модели | Потеря качества |
|---|---|---|---|
| FP32 | 32 | ~12 GB | Нет (оригинал) |
| FP16 | 16 | ~6 GB | Минимальная |
| INT8 (Q8) | 8 | ~3 GB | Очень малая |
| **Q4_K_M** | ~4.5 | **~2 GB** | Малая |
| Q2 | 2 | ~1 GB | Заметная |

### Как это работает (упрощённо)

FP32 может представить число с точностью до 7 знаков: `0.1234567`.
Q4 может представить ~16 уровней (2^4). Вместо точного числа модель хранит
«ближайший уровень»: `0.125`.

Это как сжатие JPEG: фотография немного теряет в качестве, но весит в 10 раз
меньше. Для текста потеря обычно незаметна.

### Q4_K_M — что значит название

- **Q4** — 4-битная квантизация.
- **K** — k-quants: более умный метод, где разные слои квантизуются по-разному
  (важные слои сохраняют больше бит).
- **M** — medium: средний баланс между размером и качеством (есть S — small, L — large).

### В нашем проекте

Qwen 2.5 3B Instruct в формате **Q4_K_M GGUF**:
- Оригинал: ~6 GB (FP16).
- После квантизации: **~2 GB** — помещается в RAM обычного ноутбука.
- Качество ответов на наших тестах — практически не отличается от FP16.

## 16. Инференс vs обучение

**Вопрос интервьюера:** *Мы модель обучаем или используем? В чём разница?*

### Обучение (training)

- **Цель**: модель учит веса, чтобы хорошо предсказывать.
- **Нужны**: размеченные данные (вопрос + правильный ответ), GPU, часы/дни.
- **Процесс**: прямой проход → ошибка → обратное распространение → обновление весов.
- **Дорого**: миллионы долларов для больших моделей.

### Инференс (inference)

- **Цель**: модель с готовыми весами делает предсказание.
- **Нужны**: обученная модель + входные данные.
- **Процесс**: только прямой проход (forward pass). Веса не меняются.
- **Дёшево**: миллисекунды на CPU.

### Аналогия

- **Обучение** — школьник учится решать уравнения (годы, учебники, учителя).
- **Инференс** — взрослый решает конкретное уравнение (секунды, ручка).

### В нашем проекте

Мы делаем **только инференс**:
1. Загружаем предобученные веса (скачаны из HuggingFace / GGUF).
2. Подаём вопрос → получаем ответ.
3. Веса не изменяются, модель «не учится» на наших данных.

Это важно понимать: даже если модель ошибается на наших документах, она не станет
лучше от повторных запусков. Для улучшения нужно:
- Лучший retrieval (найти нужный фрагмент).
- Лучший промпт (для генеративного ридера).
- Дообучение на наших данных (fine-tuning — не реализовано, требует GPU).

## 17. Промпт-инжиниринг

**Вопрос интервьюера:** *Что такое промпт-инжиниринг? Почему просто «ответь на вопрос»
не работает?*

### Что такое промпт

**Промпт** — текстовая инструкция, которую мы даём языковой модели. Модель генерирует
ответ, следуя (или не следуя) этой инструкции.

### Почему это важно

LLM — это «универсальные» модели. Без чётких инструкций они могут:
- Отвечать не по документу, а из своих знаний (галлюцинация).
- Давать слишком длинный или слишком короткий ответ.
- Отвечать на другом языке.
- Не ссылаться на источники.

### Наш промпт (для Qwen)

```
Ты — ассистент, который отвечает на вопрос СТРОГО по предоставленным фрагментам документа.
Правила:
1. Используй только информацию из фрагментов ниже. Никаких внешних знаний.
2. Сначала напиши сам ответ, а В КОНЦЕ укажи номера фрагментов в [скобках].
3. НИКОГДА не отвечай одними скобками без текста.
4. Если ответа в фрагментах НЕТ — ответь ровно: НЕТ ОТВЕТА.
5. Отвечай кратко и на языке вопроса.
```

### Что мы нашли (итерации промпта)

| Проблема | Причина | Решение в промпте |
|---|---|---|
| Qwen отвечал `[2]` без текста | Нечёткий формат | Правило 3: «НИКОГДА не отвечай одними скобками» |
| Qwen отвечал из «памяти» | Нет запрета внешних знаний | Правило 1: «Никаких внешних знаний, даже если знаешь» |
| Qwen отвечал по-русски на EN-вопросы | Русский system-prompt | Автодетекция языка: кириллица → RU-промпт, иначе EN |
| Qwen не цитировал источники | Нечёткая инструкция | Правило 2: пример формата «Столица — Париж [3].» |

### Принципы промпт-инжиниринга

1. **Будьте конкретны**: «отвечай кратко» лучше, чем «отвечай».
2. **Давайте примеры**: модель лучше понимает формат из примера, чем из описания.
3. **Явно запрещайте** нежелательное поведение: модель не угадает, чего вы не хотите.
4. **Итерируйте**: первый промпт почти никогда не идеален. Тестируйте и правьте.

## 18. Галлюцинации LLM

**Вопрос интервьюера:** *Почему языковая модель «врёт» и как с этим бороться?*

### Что такое галлюцинация

**Галлюцинация** — когда модель генерирует текст, который выглядит правдоподобно,
но фактически неверен или не основан на предоставленном контексте.

### Почему это происходит

LLM обучена предсказывать «наиболее вероятное продолжение текста». Если вопрос
звучит похоже на что-то из обучающих данных, модель может «вспомнить» и ответить
из «памяти», игнорируя контекст.

Пример из нашего проекта:
```
Вопрос: "Who founded Telenet?"
Контекст: [фрагмент про телекоммуникации, но НЕ про основателя Telenet]
Qwen ответил: "Telenet was founded by BBN Technologies."
```
Ответ правильный с точки зрения общих знаний, но **не основан на контексте** —
это галлюцинация в рамках RAG-пайплайна.

### Виды галлюцинаций

| Вид | Пример | Опасность |
|---|---|---|
| **Intrinsic** (искажение контекста) | Контекст: «кредит 14.9%», ответ: «кредит 14.5%» | Высокая — неправильный факт |
| **Extrinsic** (внешние знания) | Отвечает из обучающих данных, не из документа | Средняя — правда, но не из источника |
| **Fabrication** (выдумка) | Придумывает факт, которого нигде нет | Очень высокая |

### Как мы боремся

1. **RAG** (retrieval-augmented generation) — модель получает конкретные фрагменты
   документа, а не отвечает «из головы». Это уже снижает галлюцинации.

2. **Grounded prompt** — явный запрет внешних знаний: «используй ТОЛЬКО фрагменты ниже».

3. **Цитирование** — модель обязана указывать номера фрагментов [1], [2]. Это
   дисциплинирует: сложнее галлюцинировать, если нужно сослаться на конкретный источник.

4. **Hybrid-верификация (наш guardrail)** — после генерации Qwen, экстрактивный BERT
   проверяет, есть ли подтверждение ответа в тех же фрагментах:
   - Если BERT нашёл → `verified by extractive`.
   - Если BERT не нашёл → `UNVERIFIED (возможна галлюцинация)`.
   - Перехватывает ~70% галлюцинаций.

5. **Абстенция** — если ответа нет, лучше честно отказаться, чем выдумать.
   Двухуровневая (retrieval + reader) абстенция снижает ложные ответы.

### Важное ограничение

Ни один метод не даёт 100% защиты. Даже с hybrid-верификацией ~30% галлюцинаций
проходят. В критичных сценариях (медицина, финансы) нужна человеческая проверка.

## 19. Latency и throughput

**Вопрос интервьюера:** *Как измерять скорость системы? Что важнее — задержка или
пропускная способность?*

### Latency (задержка)

**Latency** — время от отправки запроса до получения ответа. Измеряется в
миллисекундах (мс) или секундах.

Пример: «пользователь задал вопрос → через 230 мс получил ответ». Latency = 230 мс.

### Throughput (пропускная способность)

**Throughput** — сколько запросов система обрабатывает за единицу времени.
Измеряется в запросах/секунду (QPS).

Пример: «система обрабатывает 4.3 вопроса в секунду». Throughput = 4.3 QPS.

### Связь

```
Throughput ≈ 1 / Latency    (для однопоточной обработки)
```

При latency = 230 мс → ~4.3 QPS. Если обрабатывать запросы параллельно (несколько
воркеров), throughput растёт, а latency остаётся прежним.

### Перцентили (p50, p95, p99)

Не все запросы одинаковы. Короткий вопрос обрабатывается быстрее длинного.
Поэтому измеряют **перцентили**:

- **p50** (медиана) — 50% запросов быстрее этого. «Типичный» запрос.
- **p95** — 95% запросов быстрее. «Почти худший случай».
- **p99** — 99% запросов быстрее. «Хвост» распределения.

Пример:
```
p50 = 200 мс   (типичный запрос)
p95 = 500 мс   (95% уложатся в полсекунды)
p99 = 2000 мс  (1% запросов — дольше 2 секунд, например длинные вопросы)
```

### Наши значения

| Компонент | Latency | Что определяет скорость |
|---|---|---|
| Retrieval (эмбеддинг + FAISS) | ~5 мс | Быстро: один вектор + поиск |
| Reader (BERT, 1 фрагмент) | ~50 мс | Forward pass нейросети |
| Reader (BERT, k=5 фрагментов) | ~250 мс | Линейно растёт с k (CPU) |
| End-to-end (retrieval + reader) | ~230 мс/вопрос | Reader доминирует |
| Qwen 3B (генерация) | ~10 с/вопрос | Авторегрессия: токен за токеном |

### Почему latency важнее throughput для нас

Наш чат-бот — **интерактивный**: пользователь задаёт вопрос и ждёт ответ.
Для него важна latency (хочу ответ за 0.5 с, а не за 10 с), а не throughput
(сколько пользователей одновременно — для прототипа неважно).

Для экстрактивного ридера: 230 мс — пользователь не заметит задержки.
Для генеративного: 10 с — заметно, но приемлемо для сложного вопроса.
