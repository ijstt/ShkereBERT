# Стратегия чанкинга: обоснование и параметры

## Почему чанкинг важен

1. **Окно ридера ограничено** (max_seq_len=384 токена у tinyroberta) — длинный документ не помещается.
2. **Качество поиска** — эмбеддинг короткого смыслового фрагмента точнее представляет его тему, чем эмбеддинг всего документа.
3. **Цитируемость** — ответ должен ссылаться на конкретный фрагмент, а не на весь документ.

## Выбор: sentence-aware + overlap (дефолт)

```python
ChunkConfig(
    size=256,        # целевой размер в токенах эмбеддера
    overlap=64,      # 25% перекрытия
    strategy="sentence"  # по границам предложений
)
```

### Обоснование параметров

| Параметр | Значение | Почему |
|---|---|---|
| **Токены, не символы** | 256 токенов | Модель «видит» токены; 256 токенов ≈ 180–220 слов ≈ 1–2 абзаца |
| **Sentence-aware** | Да | Сохраняет смысловые границы; ридер получает связный текст |
| **Overlap 64 (25%)** | Да | Гарантирует, что факт на границе чанков целиком попадёт хотя бы в один |
| **Fallback на токены** | Если предложение > size | Длинные предложения режутся скользящим окном |

## Альтернатива: fixed-token (для ablation)

```python
ChunkConfig(size=256, overlap=64, strategy="fixed")
```
Режет строго по токенам с шагом `size - overlap`. Используется в `eval/ablations.py`
для сравнения — на SQuAD разницы почти нет (контексты короткие), на длинных доках
sentence-aware выигрывает за счёт сохранения границ.

## Эмпирические выводы (из `eval/ablations.py` и `eval/eval_longdoc.py`)

### На коротких SQuAD-контекстах (медиана 141 токен)
- При `chunk_size ≥ 256` режется **< 8% документов** — чанкинг почти не срабатывает
- F1 практически не зависит от размера (77.3–78.1)
- **Вывод v1**: «размер чанка не важен» — был артефактом коротких текстов

### На длинных документах (LongDoc: медиана 5.3k токенов, 100% режутся)
| chunk_size | F1 | HasAns_F1 | NoAns_F1 | answer-recall@5 | latency |
|---|---|---|---|---|---|
| **64** | **74.7** | 58.1 | **88.4** | **0.875** | **181 мс** |
| 128 | 72.2 | 55.5 | 86.0 | 0.853 | 218 мс |
| 256 | 69.3 | 57.3 | 79.3 | 0.853 | 371 мс |
| 512 | 71.8 | 58.5 | 82.9 | 0.846 | 544 мс |

**Ключевые находки:**
1. **Разброс 5.4 F1** между размерами — чанкинг ВАЖЕН на реальных документах
2. **Мелкие чанки выигрывают за счёт NoAns_F1** (88.4 vs 79.3): меньше постороннего
   текста → чище null-score решение «ответа нет»
3. **Мелкие чанки быстрее** (181 vs 371 мс): ридер читает меньше токенов на чанк
4. **Внутренние дистракторы** — главная сложность: doc-recall@5 0.95–0.98, но
   answer-recall@5 только 0.85–0.88 — поиск *внутри* статьи теряет 10–15%

## Рекомендация для продакшена

| Тип документов | chunk_size | overlap | strategy |
|---|---|---|---|
| Короткие (FAQ, регламенты < 1 стр) | 256 | 64 | sentence |
| **Длинные (PDF, статьи, контракты)** | **64–128** | **25%** | **sentence** |
| Очень длинные / шумные | 64 | 16 | sentence |

**Правило**: начните с 64–128 токенов для реальных документов; калибруйте τ на своих данных.

    counter = TokenCounter(embed_model)
    if cfg.strategy == "fixed":
        texts = _split_by_tokens(doc.text, counter, cfg.size, cfg.overlap)
    else:
        texts = _chunk_sentence_aware(doc.text, counter, cfg.size, cfg.overlap)
