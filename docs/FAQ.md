## 1. Как объяснить пайплайн проекта?

Лучший способ — пройти по цепочке на конкретном примере. Порядок:

1. **Загрузка документа** (`ingest.py`) — PDF, TXT или SQuAD v2.
2. **Чанкинг** (`chunking.py`) — режем на фрагменты ~256 токенов с overlap 64.
3. **Эмбеддинги** (`embeddings.py`) — каждый чанк → вектор 384 чисел (MiniLM).
4. **Индекс** (`index.py`) — складываем в FAISS для быстрого поиска.
5. **Поиск** (`retriever.py`) — вопрос → вектор → top-5 ближайших чанков.
6. **Ридер** (`reader.py` / `generator.py`) — извлекает/генерирует ответ.
7. **Абстенция** (`pipeline.py`) — если ответа нет → честно говорим.
8. **Ответ + цитата** — пользователь видит ответ и фрагмент-источник.

Совет: откройте `pipeline.py`, метод `answer()` — там вся логика в 15 строках.
Пройдите сверху вниз и объясните каждый шаг.

```
Документ → чанкинг → эмбеддинги → FAISS → retrieval top-5 → ридер → абстенция → ответ + цитата
```

## 2. Что такое ползунок «Контекстов SQuAD» в Streamlit?

Это количество уникальных параграфов из SQuAD v2 validation, которые загружаются как
«документная база» для демо.

- **50** — быстро, но мало документов.
- **200** (дефолт) — баланс для демо.
- **1000** — почти весь корпус, максимальное покрытие, но индексация дольше.

По сути демо-режим: вместо вашего PDF система берёт параграфы Wikipedia из SQuAD и
строит поисковый индекс. Можно задавать вопросы типа «What year was the Battle of
Normandy?» и получать ответы с цитатами.

## 3. Как работают три режима ридера?

### Extractive (BERT)
BERT получает пару (вопрос, фрагмент) и **выделяет кусок текста** (span) — начало
и конец ответа прямо в тексте. Быстро (~50 мс), точно (F1 75.8), но только один
непрерывный кусок из одного фрагмента.

### Generative (Qwen)
Локальная LLM получает найденные фрагменты и **генерирует ответ в свободной форме**
с цитатами [1], [2]. Может собрать информацию из нескольких фрагментов, работает на
русском. Медленнее (~10 с на CPU), может галлюцинировать.

### Hybrid (Qwen + BERT-верификатор)
Qwen генерирует ответ, затем BERT проверяет — есть ли подтверждение в тех же
фрагментах. Если BERT подтвердил → «verified». Если нет → «UNVERIFIED (возможна
галлюцинация)». Перехватывает ~70% галлюцинаций.

## 4. Есть ли у нас своя модель? Мы что-то обучали?

**Нет, мы ничего не обучали.** Используем три предобученные модели «из коробки»:

| Модель | Кто обучил | Наше использование |
|---|---|---|
| `all-MiniLM-L6-v2` | Microsoft | Эмбеддинги для поиска |
| `tinyroberta-squad2` | deepset | Экстрактивный QA |
| `Qwen 2.5 3B` | Alibaba | Генеративный QA |

**Наша работа** — пайплайн вокруг них: чанкинг, индекс, retrieval, абстенция,
калибровка порога, hybrid-верификация, оценка, интерфейсы. Модели — «кирпичи»,
мы построили «здание».

Для обучения своей модели нужны GPU и большой размеченный датасет. Transfer learning
позволяет использовать чужое предобучение без затрат — это стандартный подход в
индустрии.

## 5. Как работает цитирование источников?

### Extractive (BERT)
BERT вырезает span из конкретного чанка. Мы знаем id чанка → помечаем его как
`is_answer_source = True` в списке источников.

### Generative (Qwen)
Промпт обязывает Qwen ставить номера фрагментов в скобках: `«Столица — Париж [3].»`
После генерации парсим номера регуляркой:
```python
idxs = re.findall(r"\[(\d+)\]", text)  # [1], [2], [3]...
cited_ids = [chunks[i - 1].id for i in idxs]
```
Все процитированные чанки помечаются `is_answer_source = True`.

## 6. Как работает верификация (hybrid-режим)?

Пошагово (`pipeline.py`, метод `_answer_hybrid`):

1. **Qwen генерирует** ответ по найденным фрагментам.
2. **BERT читает те же фрагменты** и ищет span-ответ.
3. Проверяем BERT: `gap = null_score - best_span_score`.
   - `gap <= tau` и BERT нашёл текст → подтверждает.
   - `gap > tau` → не подтверждает.
4. Решение:
   - Qwen ответил + BERT подтвердил → `"verified by extractive"`.
   - Qwen ответил + BERT не подтвердил → `"UNVERIFIED (возможна галлюцинация)"`.
   - Qwen отказался + BERT нашёл → отдаём ответ BERT (fallback).
   - Оба отказались → «нет ответа».

BERT — «детектор лжи»: не генерирует текст, а проверяет, есть ли ответ в фрагментах.

## 7. Для чего каждая из трёх моделей?

### all-MiniLM-L6-v2 — «поисковик»
Превращает текст в вектор (384 числа). Вопрос → вектор, чанки → векторы. FAISS
ищет ближайшие. **Модель НЕ читает ответ** — она только ищет, где ответ может быть.

### tinyroberta-squad2 — «читатель»
Получает пару (вопрос, фрагмент). Для каждого токена выдаёт: «это начало ответа?»
и «это конец ответа?». Вырезает span. Также выдаёт null_score для абстенции.
**Модель НЕ ищет фрагменты** — она читает то, что нашёл MiniLM.

### Qwen 2.5 3B — «генератор»
Получает все найденные фрагменты + промпт и генерирует ответ токен за токеном.
Может объединить факты из нескольких фрагментов. Работает на русском.

```
Вопрос → MiniLM (ПОИСК) → tinyroberta (ЧТЕНИЕ) или Qwen (ГЕНЕРАЦИЯ) → Ответ
```

## 8. Как Qwen ищет ответ в тексте?

Qwen **не ищет** в классическом смысле — он **генерирует** текст токен за токеном
(авторегрессия). Получает промпт с пронумерованными фрагментами и вопросом. Благодаря
attention «видит» все фрагменты одновременно и может найти нужную информацию, объединить
факты из разных мест, перефразировать.

Промпт ограничивает: «отвечай ТОЛЬКО по фрагментам». Без этого Qwen отвечал бы из
своих обучающих данных.

Ключевая разница: BERT **вырезает** кусок текста как есть. Qwen **создаёт** новый текст.

## 9. Как работает разбивка на чанки?

**Стратегия: sentence-aware чанкинг** (`chunking.py`):

1. **Разбиваем текст на предложения** (регуляркой по `.!?`).
2. **Жадно упаковываем** предложения в чанки до ~256 токенов.
3. **Overlap** — при переходе к новому чанку оставляем «хвост» из последних
   предложений (~64 токена), чтобы ответ не разрезался на границе.
4. **Длинные предложения** (>256 токенов) принудительно режутся скользящим окном.

**Почему в токенах, а не в символах?** Модель видит токены. Слово
«internationalization» — 1 слово, но ~5 токенов. Считая в токенах, мы точно знаем,
сколько «увидит» модель.

Выбор 256 обоснован ablation-исследованием: 128 — лучший recall, но больше чанков;
512 — чанки «шумят»; 256 — баланс.

## 10. Как проект работает для русского языка?

Для русского работает **только генеративный режим (Qwen)**:

| Компонент | Английский | Русский |
|---|---|---|
| **Эмбеддер** | `all-MiniLM-L6-v2` | `multilingual-MiniLM-L12-v2` |
| **Extractive** | Работает | Не работает (модель EN-only) |
| **Generative** | Работает (EN-промпт) | Работает (RU-промпт) |
| **Hybrid** | Полноценно | Qwen работает, BERT-проверка бесполезна |

Переключение: пресет `multilingual_config()` в `config.py` меняет эмбеддер на
мультиязычный. Промпт Qwen переключается автоматически по языку вопроса.

## 11. Как мы определяем язык вопроса?

Проверяем наличие кириллицы регуляркой:

```python
_CYRILLIC = re.compile(r"[а-яё]", re.IGNORECASE)

if _CYRILLIC.search(question):
    # Русский → RU-промпт, маркер "НЕТ ОТВЕТА"
else:
    # Английский → EN-промпт, маркер "NO ANSWER"
```

Влияет только на генеративный ридер (выбор system-промпта и маркера отказа).

## 12. Как происходит матчинг вопрос-ответ для каждого ридера?

Общий путь (одинаков для всех):
1. Вопрос → эмбеддинг (MiniLM) → FAISS ищет top-5 чанков.
2. Проверка `dense_score < min_score` → если да, сразу «нет ответа».
3. Передаём 5 чанков ридеру.

Дальше расходятся:

**Extractive**: BERT обрабатывает каждый чанк **отдельно**. Для каждого токена выдаёт
start/end логиты. Лучший span = `argmax(start[i] + end[j])`. Проверяет gap vs tau.

**Generative**: Qwen получает **все 5 чанков сразу** в одном промпте. Генерирует ответ
токен за токеном. Может собрать из нескольких фрагментов. Парсим цитаты [n].

**Hybrid**: Qwen генерирует + BERT читает те же чанки. Если BERT подтвердил (gap ≤ tau)
→ verified. Если нет → UNVERIFIED.

## 13. С какими проблемами столкнулись и что решили?

| Проблема | Решение |
|---|---|
| Бесконечный цикл в чанкере (длинное предложение + overlap) | Переписали на for-цикл с гарантией продвижения |
| Утечка при калибровке tau (F1 завышен на 2.5) | Разделили calibration/test |
| Qwen отвечал из «памяти» (галлюцинации на adversarial) | Явный запрет внешних знаний в промпте + hybrid-верификация |
| Qwen отвечал одними скобками `[2]` без текста | Добавили запрет в промпт + пример формата |
| Русский промпт ломал ответы на EN-вопросы | Автодетекция языка по кириллице |
| sentence-transformers 5.x ломал офлайн-загрузку | Пин `<5` в requirements |
| Батчинг ридера на CPU не ускорил | Вернулись к пофрагментной обработке (паддинг добавлял пустой compute) |

## 14. Какие изъяны у проекта?

1. **Extractive ридер** — только один непрерывный span из одного фрагмента.
2. **Домен** — метрики на SQuAD (Wikipedia). На узких доменах нужна перекалибровка tau.
3. **Qwen на CPU** — ~10 с/вопрос (на GPU было бы ~0.5 с).
4. **PDF** — только текстовое извлечение (pypdf). Сканы, таблицы — не работают.
5. **Русский** — только generative. Hybrid-проверка бесполезна (BERT не понимает RU).
6. **tau не переносится** — при смене chunk_size/домена tau* меняется кардинально.
7. **Контексты SQuAD короткие** (медиана 141 токен) — ablation по chunk_size
   малоинформативен.

## 16. Почему медиана токенов SQuAD — 141?

Мы **замерили**. SQuAD v2 — параграфы из Wikipedia. Посчитали длину всех уникальных
контекстов в токенах (через токенизатор эмбеддера) — медиана ~141 токен.

Это значит: при `chunk_size=256` более 90% контекстов **вообще не режутся** — влезают
целиком. Поэтому ablation по chunk_size на SQuAD малоинформативен. Настоящую разницу
увидели на `eval_longdoc.py` (склеенные статьи, медиана ~5300 токенов) — там chunk_size
даёт 5 пунктов F1.

## 17. Как разделить обязанности в команде из 3 человек?

| Участник | Роль | Что делал |
|---|---|---|
| **Участник 1** | ML-инженер / бэкенд | Ядро пайплайна: `chunking.py`, `embeddings.py`, `index.py`, `retriever.py`, `reader.py`, `pipeline.py`, `config.py` |
| **Участник 2** | Генеративная часть + оценка | `generator.py`, промпт-инжиниринг, hybrid-режим. Все eval-скрипты, калибровка tau, multi-seed, графики |
| **Участник 3** | Данные + интерфейсы + документация | `ingest.py` (SQuAD/PDF/TXT), CLI, Streamlit UI, документация, тесты, `run.sh` |

На защите:
- **Участник 1**: архитектура, чанкинг, эмбеддинги, retrieval, экстрактивный ридер.
- **Участник 2**: генеративный ридер, hybrid, метрики, ablation — показывает графики.
- **Участник 3**: демо (Streamlit/CLI), загрузка данных, ограничения.
