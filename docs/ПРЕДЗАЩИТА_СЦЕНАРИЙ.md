# Предзащита: что и как рассказывать

Формат: ~7–10 минут рассказа + вопросы. Аудитория двойная — комиссия (методология,
метрики, самостоятельность) и куратор из банка (инженерная зрелость, применимость).
Хорошая новость: одна и та же история закрывает обоих, если правильно расставить акценты.

---

## Сюжет выступления (структура на 8 минут)

### 1. Задача и рамка (1 мин)
«Чат-бот по документу: вопрос → ответ по найденному фрагменту, с источником, и честное
"нет ответа", когда ответа в документе нет». Сразу назвать два усиления рамки, которые
сделали сами: (а) целевой сценарий — **on-premise ассистент по внутренним документам**
(мотив: практика в банке, документы не должны покидать периметр); (б) качество не
демонстрируется примерами, а **доказывается метриками** по официальному протоколу SQuAD v2.

### 2. Архитектура за 90 секунд (схема из ПОСОБИЕ.md §2)
Документ → чанкинг по предложениям (в токенах, с overlap) → MiniLM-эмбеддинги → FAISS →
top-k → ридер. Ридер — **три режима на выбор**: extractive (BERT-span, быстрый, метрики),
generative (локальная Qwen, естественные ответы, русский), hybrid (Qwen отвечает, BERT
верифицирует — анти-галлюцинационный контроль). Абстенция двухуровневая: «вопрос не по
документу» ловится на поиске, «ответа нет в документе» — на ридере с калиброванным порогом.

Здесь одна фраза для куратора: «всё на CPU, полностью офлайн — прогоны оценки идут с
физически выключенной сетью, то есть on-premise-пригодность подтверждена буквально».

### 3. Главные числа (1 мин, слайд с таблицей)
- End-to-end на held-out test: **EM 73.6 / F1 75.8**, отказ на вопросах без ответа
  **NoAns_F1 84.6**, ~0.3 с/вопрос на CPU. По 5 сидам: **F1 76.9 ± 3.9** (95% CI).
- Поиск: **Recall@5 0.94, MRR 0.86**.
- Oracle-потолок ридера **F1 81.0** → цена этапа поиска — **5.3 пункта F1** (измерена,
  не предположена).
- Режимы ридера (одна выборка, продуктовый путь): extractive F1 76.0 / containment 0.93 /
  0.28 с; generative — containment 0.89, но строгий F1 43.8 (штраф за многословие измерен)
  и 10.4 с; hybrid-верификатор помечает **70% галлюцинаций** на вопросах без ответа.

### 4. Три истории, которые отличают работу от «собрал туториал» (3 мин — ядро!)

**История 1: нашли утечку в собственной оценке.** В первой версии порог абстенции τ
калибровался и отчитывался на одной выборке. Разделили calibration/test — честный F1
оказался на ~2.5 пункта ниже. Мораль, проговорить дословно: «мы предпочли ухудшить свои
цифры, но сделать их честными». Комиссия такое запоминает.

**История 2: эксперимент, который ничего не мерил, и как мы его починили.** Ablation по
размеру чанка показывал «размер не влияет». Копнули: SQuAD-контексты короткие (медиана
141 токен), при chunk≥256 режется <8% документов — эксперимент не тестировал чанкинг
вообще. Построили честный стенд: склеили статьи в документы по 5–6 тыс. токенов (режется
100%), ввели метрику answer-recall@k, ловящую разрезание ответа границей. Показать график
`longdoc_chunking.png` и вывод по нему.

**История 3: интуиция с GPU не переносится на CPU.** Батчинг ридера (один forward на k
фрагментов) на CPU оказался *медленнее* пофрагментной обработки — паддинг до длиннейшего
фрагмента добавляет пустой compute. Замерили, откатили, зафиксировали в CHANGELOG. Для
on-premise на обычном железе такие микрорешения и складываются в латентность.

**История 4 (запасная): количественная оценка отдебажила LLM.** Прогон оценки генерации
нашёл две проблемы, которые «на глаз» не видны: русский system-prompt заставлял Qwen
отвечать по-русски на английские вопросы, а на adversarial-вопросах без ответа модель
отвечала из собственной памяти (Telenet, Galileo — фактов нет во фрагментах). Первую
починили языко-адаптивным промптом, вторую — закрыли верификатором (ловит 70%) и честно
записали в ограничения. Мораль: генеративный компонент без метрик не дебажится.

(Если время поджимает — рассказывать 1 и 2, остальные держать для вопросов.)

### 5. Живое демо (1 мин, страховка — скриншоты)
RU-сценарий на банковском демо-документе, режим hybrid или generative:
- «Какой минимальный взнос по вкладу?» → «500 рублей [1]» + источник;
- составной вопрос (взнос + ставка) → ответ собран из **двух** фрагментов с цитатами —
  то, что extractive не умеет в принципе;
- вопрос не по документу → «В документе нет ответа» (показать, что система не выдумывает).

### 6. Ограничения и план (30 сек)
Назвать самим (выглядит сильно): extractive-ридер только EN; абстенция генератора — промпт,
не калиброванный порог; SQuAD ≠ банковские документы, перенос требует своей разметки.
План: hybrid BM25 + reranker, доменная калибровка, OCR-слой. Финальная фраза: «прототип
готов к пилоту на реальном корпусе — архитектура переносится без переделки».

## Акценты под куратора (вплетать, не выделять в отдельный блок)

1. **Язык рисков, не технологий**: цитаты-источники = проверяемость; калиброванный отказ =
   управляемая цена ошибки; UNVERIFIED-пометка в hybrid = контроль галлюцинаций. Банк
   покупает не «RAG», а снижение риска дать клиенту неверный тариф.
2. **Стоимость владения**: open-source стек, одна CPU-машина, нулевые лицензии и нулевая
   плата за токены; качество поднимается точечно (reranker, крупнее ридер), когда/если
   появится железо.
3. **Инженерная культура**: CHANGELOG с историей решений, воспроизводимые сиды, результаты
   в git как доказательная база, тесты, поймавшие реальные баги. Это сигнал «может работать
   в команде», который куратор считывает лучше любых слов.
4. **Мостик к geoanalytics** (если спросят про опыт): тот же подход — локальный каскад
   малых моделей под ограниченное железо; ShkereBERT — тот же принцип, доведённый до
   измеримого качества по официальной метрике.

## Чего НЕ делать

- Не начинать с моделей («мы взяли MiniLM и tinyroberta…») — начинать с задачи и рамки.
- Не зачитывать все цифры из таблиц — три числа наизусть (75.8 / 84.6 / 0.94), остальное
  «в eval/results, могу показать».
- Не защищаться на вопросах про ограничения — соглашаться и показывать, что ограничение
  уже описано и запланировано (ссылка на Блок 5 вопросника).
- Не говорить «мы попробовали и заработало» — везде заменять на «мы измерили: …».
